# Melo TUI 播放热路径与可用性修复设计

日期：2026-04-14

## 1. 背景

当前 Melo 的 TUI 在“能用”与“顺手”之间还隔着一层明显的状态设计问题，主要体现在以下五类现象：

- 歌单预览似乎只能看到约 `20` 首，无法稳定滚动到后续条目
- 歌词不会随着播放时间自动滚动
- 封面没有稳定展示，尤其没有把“音频内嵌封面”作为正式来源信息对外表达
- `melo --verbose` 进入 TUI 后，日志仍可能继续输出到当前终端并覆盖界面
- 在歌单预览里按 `Enter` 切歌时，确认路径体感卡顿，底部时间与进度也可能停住

这些现象并不是五个彼此独立的 bug，而是同一套问题的不同表现：

- 当前切歌确认仍然绑在偏重的聚合快照链路上
- TUI 已经有 runtime snapshot、lyrics follow state、track content cache 等雏形，但还没有真正接成“高频真相 + 低频内容 + 本地展示态”三层
- 界面时间展示仍然过度依赖 daemon 的周期性位置更新，没有把 TUI 作为平滑显示层

因此，这轮不采用“逐项打补丁”的思路，而采用一次中等重构，把 TUI 的播放热路径、预览滚动、歌词跟随、封面来源表达和 verbose/TUI 边界一起收敛。

## 2. 目标

本次设计目标如下：

- 让歌单预览能够展示当前数据库中的完整已知内容，并支持稳定滚动访问全部条目
- 对后台补扫中的目录歌单，允许预览随着扫描进度持续增长
- 让歌词区支持：
  - 当前句高亮
  - 自动跟随播放时间滚动
  - 手动浏览后默认 `3s` 自动恢复跟随
- 让封面区稳定表达当前曲是否有封面，以及封面来自哪里
  - sidecar 图片
  - 音频内嵌封面
  - 无封面
- 让 `Enter` 切歌继续以 daemon 为真相源确认，但确认路径显著变轻，不再等待整页低频内容一起返回
- 让底部时间与进度主要由 TUI 本地时钟平滑推进，daemon 负责纠偏
- 让 `melo --verbose` 在进入 TUI 前保留启动阶段日志，在进入 alternate screen 后自动静音终端输出，避免覆盖界面

## 3. 非目标

本次明确不做：

- 不处理 Swagger 地址限时 `403`
- 不处理 CLI / daemon 彩色输出增强
- 不在本次实现 TUI 内日志面板
- 不强制在终端中渲染真正的位图封面
- 不重写整个 TUI 架构为全新前端模型
- 不删除现有 `/api/ws/tui`、`/api/tui/home` 等兼容链路

以上内容应作为后续单独的“观测与文档体验”子项目处理，不与本次 spec 混合。

## 4. 方案对比

### 4.1 方案 A：补丁式修复

- 保留当前 `TuiSnapshot` 主链路
- 分别修滚动、歌词跟随、封面文案和 verbose 静音

优点：

- 改动最小
- 上手最快

缺点：

- `Enter` 切歌卡顿无法根治
- 进度平滑展示与歌词跟随仍然容易受重快照牵制
- 多处状态来源继续混杂

### 4.2 方案 B：分层收口

- 保留现有总体运行栈
- 将 TUI 明确拆成“高频运行时真相 + 低频内容查询 + 本地展示态”
- 将切歌确认与低频内容彻底分开

优点：

- 能同时解决 5 个问题
- 改动范围仍控制在中等重构
- 与后续长期方向一致

缺点：

- 需要调整 TUI 本地状态与部分 API 使用方式
- 测试面会比补丁式方案更广

### 4.3 方案 C：彻底新模型

- 让新 TUI 基本不再依赖旧聚合快照
- 直接切到 bootstrap + runtime WS + 低频查询的全新客户端模型

优点：

- 模型最干净

缺点：

- 风险明显过大
- 超出本轮修复范围

### 4.4 结论

本次采用 **方案 B：分层收口**。

## 5. 总体架构

### 5.1 三层状态模型

TUI 侧状态明确拆成三层：

1. `daemon 高频真相`
2. `低频内容查询`
3. `TUI 本地展示态`

### 5.2 daemon 高频真相

daemon 继续作为播放真相源，但高频链路只负责真正需要远端确认的字段：

- `generation`
- `playback_state`
- `current_source_ref`
- `current_song_id`
- `current_index`
- `position_seconds`
- `duration_seconds`
- `last_error`

这一层继续通过轻量 runtime WebSocket 提供。

### 5.3 低频内容查询

以下内容统一视为低频内容，不进入切歌热路径：

- 歌单完整预览
- 当前曲歌词时间轴
- 当前曲封面来源信息
- 当前曲详情补充信息

### 5.4 TUI 本地展示态

TUI 会话内本地状态负责：

- 左侧歌单选中态
- 右侧预览选中态
- 歌单预览视口与滚动位置
- 歌词视口与跟随状态
- 切歌 pending 态
- 本地时间锚点
- 低频内容缓存

## 6. 数据与组件边界

### 6.1 预览数据模型

歌单预览以“当前数据库中的完整已知内容”为准。

规则如下：

- 不做固定条数分页
- 第一次查询返回当前数据库里已有的完整歌单成员
- 若后台补扫继续写入该歌单，后续重新查询应返回增长后的完整结果

### 6.2 歌词内容模型

歌词正式展示使用 `TrackContentSnapshot` 中的解析后时间轴：

- `song_id`
- `lyrics`
  - `timestamp_seconds`
  - `text`
- `refresh_token`

`TuiSnapshot.current_track.lyrics` 仅作为低频内容未到达前的临时 fallback，不作为正式滚动和高亮依据。

### 6.3 封面内容模型

封面区统一显示来源信息，而不是仅靠图片协议探测决定“有/无封面”。

本次对用户可见的三种状态固定为：

- `sidecar`
  - 显示封面来自外置图片，并显示路径
- `embedded`
  - 显示封面来自音频元数据
- `none`
  - 显示无封面

如果终端本身支持图片协议，后续可以在该模型之上再做增强，但本次 spec 不以此为前提。

### 6.4 本地时间模型

新增一份仅属于 TUI 的本地时间锚点状态，至少包含：

- `anchor_position_seconds`
- `anchor_received_at`
- `playback_state`
- `duration_seconds`
- `generation`

它不是真相源，只用于平滑展示当前时间与进度条。

## 7. 歌单预览与滚动设计

### 7.1 完整预览策略

`/api/playlists/preview` 继续作为预览数据入口，但语义固定为：

- 返回该歌单当前完整预览
- 不允许内部隐藏固定 limit

### 7.2 渐进增长策略

对于 direct-open 目录歌单：

- 后台补扫期间，TUI 允许先看到已知部分
- TUI 继续通过低频首页快照或等价的低频来源更新感知歌单 `count`
- 当当前歌单 `count` 变化时，TUI 应后台重拉一次该歌单预览
- 新内容应直接追加进入当前预览，而不是保持在旧长度

如果某目录此前已经完整扫描并写入数据库，则首次预览就应直接返回数据库中的完整 100+ 首内容。

### 7.3 视口与滚动

右侧预览区不再只是把字符串拼成 `Paragraph`，而要具备正式的列表视口行为：

- 每帧根据面板高度更新 `visible_height`
- 选中项变化时调用 `follow_selection`
- 滚轮、上下键、PageUp、PageDown、Home、End 都作用于统一的预览视口

屏幕一次只能看到有限行是正常行为；无法访问窗口之外的条目是不允许的行为。

## 8. 歌词区设计

### 8.1 内容获取

当 `current_song_id` 变化时：

- TUI 异步请求 `/api/tracks/content`
- 成功后将结果写入 `track_content_cache`
- 若缓存命中且 `refresh_token` 未变化，则直接复用

### 8.2 高亮逻辑

当前歌词高亮行由：

- 本地显示时间
- 歌词时间轴

共同计算得出。

daemon 不负责推送“当前歌词行”，TUI 自己根据本地时间定位当前句。

### 8.3 跟随状态机

歌词区至少包含三种显式状态：

- `FollowCurrent`
- `ManualBrowse`
- `ResumePending`

行为规则：

- 默认进入 `FollowCurrent`
- 用户手动滚动歌词后进入 `ManualBrowse`
- 停止滚动后开始计时，进入 `ResumePending`
- 默认 `3s` 后自动恢复 `FollowCurrent`

### 8.4 显式动作

提供一个明确的“回到当前歌词”动作：

- 取消手动浏览状态
- 立即将视口滚回当前高亮行
- 恢复自动跟随

## 9. 封面区设计

### 9.1 目标

本次封面区的核心不是“显示终端图片”，而是“稳定表达封面是否存在以及来源是什么”。

### 9.2 展示规则

固定输出规则如下：

- 外置封面存在时：
  - 显示来源类型与路径
- 没有外置封面但存在内嵌封面时：
  - 显示“封面来自音频元数据”
- 两者都不存在时：
  - 显示“无封面”

### 9.3 与现有协议探测的关系

当前 `cover protocol` 探测逻辑保留，但只能作为附加能力，不再决定“封面有没有被正确识别”。

## 10. 切歌热路径设计

### 10.1 当前问题

当前 `Enter` 切歌路径是：

- TUI 调 `/api/playlists/play`
- daemon 替换队列并返回整份 `TuiSnapshot`
- `TuiSnapshot` 又包含歌单、歌词、封面等低频内容

这会把切歌确认与整页低频查询绑在一起。

### 10.2 新确认路径

`Enter` 切歌后，TUI 只依赖 runtime WS 的高频确认，不等待整份低频内容一起返回。

确认成功的标准固定为：

- `generation` 变化
- `current_song_id` 变为目标歌曲
- `playback_state = playing`

在确认到来前：

- 预览选中保持新位置
- 当前播放区保持旧歌
- 界面仅显示轻量 pending 状态

也就是说，本次不采用乐观“先假装切到了新歌”的策略。

### 10.3 API 兼容策略

现有 `/api/playlists/play` 可继续保留为兼容入口，但本次实现必须把“播放命令提交”与“低频聚合返回”拆开。

实现约束固定如下：

- TUI 主循环不得同步等待重返回体才继续刷新界面
- daemon 播放命令路径不得为了响应本次切歌而额外重建整份低频 `TuiSnapshot`
- 热路径需要落到轻量命令确认模型
  - 可以是新的轻量播放命令接口
  - 也可以是把现有播放接口收敛为轻响应

本次要求的硬约束不是接口名字，而是“切歌命令的提交和确认不能再绑在整页重快照上”。

## 11. 本地时间与进度设计

### 11.1 基本原则

时间与进度的展示由 TUI 本地时钟主导，daemon 负责纠错。

### 11.2 时间推进

当 runtime WS 推来新的有效播放位置时，TUI 记录：

- `anchor_position_seconds`
- `anchor_received_at`
- `generation`

之后只要当前状态仍为 `playing`，界面时间按如下方式推进：

- `display_position = anchor_position_seconds + (now - anchor_received_at)`

### 11.3 纠偏规则

daemon 后续推来新的 `position_seconds` 时：

- 若与本地显示时间误差小于阈值，则不立即重置
- 若误差超过阈值，则更新锚点进行纠偏

这样既能保持平滑，也能避免本地时间长期漂移。

### 11.4 冻结规则

以下状态下本地时间不得继续推进：

- `paused`
- `stopped`
- `idle`
- `error`

切歌确认前也不得切到新歌时间。

## 12. verbose 与 TUI 边界

### 12.1 行为目标

`melo --verbose` 应保留进入 TUI 前的排障价值，但一旦进入 TUI，不能再污染当前终端界面。

### 12.2 行为规则

进入 TUI 前：

- 继续输出启动阶段日志

成功进入 alternate screen 后：

- 终端日志自动静音
- daemon 日志继续写文件
- 不再向当前 stdout/stderr 直接打运行期日志

### 12.3 兼容策略

本次不做 TUI 内日志面板。

运行期详细日志仍通过日志文件与后续命令查看。

## 13. 错误处理

### 13.1 切歌失败

若 `Enter` 切歌失败：

- 当前播放区保持旧歌
- 预览区保留用户当前选中
- 清除 pending 状态
- 用稳定错误文案提示失败原因

### 13.2 歌词或封面加载失败

若低频内容获取失败：

- 不影响播放与进度
- 歌词区显示错误态或 fallback 文案
- 封面区显示“封面信息不可用”

### 13.3 runtime 流异常

若 runtime WS 暂时断流：

- TUI 最多允许短暂保守续跑
- 随后停止本地推进并标记等待同步
- 不允许在失去真相源后长期盲跑

### 13.4 补扫中的预览刷新

预览刷新失败时：

- 保留旧预览内容
- 不重置当前选中
- 不影响当前播放

## 14. 测试策略

### 14.1 歌单预览测试

验证：

- 预览接口返回 100+ 条时，TUI 能滚动访问尾部
- 歌单 `count` 增长后，TUI 会触发重拉并展示新增条目
- 不再出现“只能看到前 20 条”的固定上限现象

### 14.2 热路径测试

验证：

- `Enter` 后主循环不等待低频内容才继续刷新
- 只有 runtime 确认到来后才切换当前播放标题与时间基线
- 切歌失败时不会错误切换到目标曲

### 14.3 本地时间测试

验证：

- `playing` 时显示时间连续递增
- `paused/stopped/error` 时冻结
- daemon 新位置到来时按阈值纠偏

### 14.4 歌词测试

验证：

- 当前句高亮与本地显示时间一致
- 手动滚动后暂停自动跟随
- 默认 `3s` 后恢复自动跟随
- 显式“回到当前歌词”动作正确恢复视口

### 14.5 封面测试

验证：

- sidecar 封面显示来源路径
- embedded artwork 显示“来自音频元数据”
- 无封面显示稳定 fallback

### 14.6 verbose 测试

验证：

- `melo --verbose` 在进入 TUI 前输出启动阶段日志
- 成功进入 TUI 后不再有日志覆盖界面

## 15. 验收标准

- 歌单预览能访问当前数据库里的完整已知内容，而不是卡在固定可见上限
- 对补扫中的目录歌单，预览会随着扫描结果继续增长
- 歌词区支持自动跟随、当前句高亮、手动浏览后默认 `3s` 恢复
- 封面区能稳定表达 sidecar、embedded、none 三种来源状态
- `melo --verbose` 不再覆盖 TUI
- `Enter` 切歌不再等待整页低频内容一起返回
- 底部时间与进度主要由 TUI 平滑推进，daemon 负责纠偏

## 16. 结论

这轮修复的核心不是“补几个交互细节”，而是把 TUI 的职责重新摆正：

- daemon 负责高频播放真相
- 低频内容通过独立查询获取
- TUI 负责平滑展示、滚动与短时本地时间推进

在这个前提下，歌单展示不全、歌词不跟随、封面没信息、verbose 覆盖界面、切歌卡顿和进度停更，都会收敛为一套一致的实现方向，而不是继续作为分散补丁存在。
