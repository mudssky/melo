# Melo 下一阶段：播放会话持久化恢复设计

日期：2026-04-10

## 1. 目标

在运行时事件和进度同步建立之后，让 daemon 重启后能够恢复上一轮播放会话。

本阶段的核心目标是：

- 持久化当前队列、当前索引和当前位置
- 在 daemon 启动时恢复可继续使用的播放会话
- 让恢复行为安全、可解释，而不是盲目自动播放
- 保持持久化失败不会污染实时播放链路

一句话概括：

把“当前播放上下文”从纯内存态推进到“可恢复但不脏”的 daemon 状态。

## 2. 当前现状与问题

当前播放器的运行时状态仍然完全驻留在内存：

- queue 只在进程内存在
- current index 不可恢复
- playback state 重启后丢失
- position 也不会被保存

这意味着：

- daemon 重启后，用户需要重新构建当前播放现场
- 自动下一首和进度同步的成果无法跨进程延续
- “本地音乐工具”体验仍缺少一个很关键的日常使用闭环

## 3. 备选方向与推荐

### 3.1 方案 A：写入单个 JSON 会话文件

优点：

- 上手快
- 实现成本低

缺点：

- 与现有数据库体系割裂
- 后续查询、迁移和一致性控制都更弱
- 更容易出现文件损坏与并发写问题

### 3.2 方案 B：写入 SQLite 会话表

优点：

- 与项目的 `SeaORM` / migration 体系一致
- 易于做迁移和演进
- 可以更清晰表达“会话头 + 队列项”结构

缺点：

- 需要新增 migration 和 repository
- 前期结构设计要更谨慎

### 3.3 方案 C：把整个会话序列化成单条 JSON 字段写入数据库

优点：

- 落地快于结构化多表
- 仍然复用现有数据库

缺点：

- 难以做局部验证和未来演进
- 结构虽然在库里，语义仍偏“黑箱”

### 3.4 推荐结论

本阶段选择 **方案 B：写入 SQLite 会话表**。

原因：

- 这是最符合 Melo 长期架构的一条路
- 后续扩展音量、模式、恢复策略时更稳定
- 便于 migration 和测试

## 4. 本阶段范围

### 4.1 纳入本阶段

- 为当前 daemon 定义单活跃会话持久化模型
- 保存队列项、当前索引、播放状态和当前位置
- daemon 启动时尝试恢复上一轮会话
- 对损坏或不完整会话做安全降级
- 为保存动作增加节流或去抖

### 4.2 不纳入本阶段

- 多设备会话同步
- 多用户会话
- 自动恢复到 `playing`
- 云端同步
- 恢复历史会话列表

## 5. 持久化模型

### 5.1 会话头

建议新增单行会话头表，表达：

- `playback_state`
- `current_index`
- `position_seconds`
- `updated_at`

建议只维护一个活动会话，不在本阶段引入历史记录。

### 5.2 会话队列项

建议新增队列表，按顺序保存：

- `position`
- `song_id`
- `path`
- `title`
- `duration_seconds`

这里建议继续使用去规范化的 `QueueItem` 结构，而不是只保存 `song_id`。

原因：

- 队列是运行时语义，不应强依赖库中歌曲记录始终存在
- 即使库记录变化，恢复逻辑仍可根据路径决定是否可播放

## 6. 恢复语义

### 6.1 daemon 启动时

启动顺序建议为：

1. 初始化配置与数据库
2. 初始化 `PlayerService`
3. 读取会话持久化状态
4. 尝试构造 `PlayerSession`
5. 发布恢复后的初始快照

### 6.2 恢复后的播放状态

推荐规则：

- 若持久化状态为 `playing` 或 `paused`，恢复后统一降级为 `stopped`
- 保留 queue、current index、position
- 等待用户显式再次 `play`

原因：

- daemon 刚启动时立即自动出声，风险较高
- 先恢复现场，再交给用户确认继续，是更稳的默认行为

### 6.3 恢复失败时

如果出现：

- 会话表损坏
- 队列项缺失
- 索引越界

则应安全降级为：

- 空队列
- `idle`
- 不向用户暴露“播放器错误”

持久化恢复失败属于 daemon 恢复问题，不应污染播放错误语义。

### 6.4 文件缺失时

如果恢复出的当前曲目文件已经不存在：

- 队列仍可恢复
- `current_index` 仍可保留
- 首次显式 `play` 时再按既有规则进入 `track_file_missing`

也就是说，恢复阶段不主动验证并破坏会话，只恢复结构化上下文。

## 7. 保存策略

### 7.1 何时保存

应在以下事件后触发保存：

- queue 变更
- current index 变更
- playback state 变更
- 进度发生明显变化

### 7.2 如何节流

推荐对位置保存做去抖，例如：

- 每 1s 到 2s 最多落一次盘

原因：

- 进度频繁更新，不宜每个 tick 都写库
- 队列或状态切换则仍应立即保存

### 7.3 保存失败时

保存失败：

- 记录 `tracing` 日志
- 不阻断播放控制
- 不把错误写进 `last_error`

这样可以把“运行时播放失败”和“后台持久化失败”严格区分开。

## 8. 模块边界

### 8.1 `src/domain/player/service.rs`

继续作为会话真相来源，负责：

- 决定何时触发保存
- 在启动时接受恢复数据并构造 session

### 8.2 `src/domain/player/session_store.rs`

建议新增独立模块，负责：

- 保存当前会话
- 读取最近会话
- 删除会话

不要把数据库细节直接塞进 `PlayerService`。

### 8.3 数据库 migration / repository

负责：

- 建表
- 读写会话记录

这一层不负责播放语义，只负责持久化映射。

## 9. API / CLI / TUI 影响

本阶段不强制新增外部命令。

对外感知主要表现为：

- daemon 重启后 `status` 仍能看到上一轮 queue 与索引
- TUI 重新连接后可直接看到恢复后的现场
- 若用户再次 `play`，会从恢复出的当前项与位置继续后续逻辑

是否支持“从恢复位置继续播”可以延后到实现计划中细化，但 spec 要求至少保留该位置数据。

## 10. 测试策略

### 10.1 存储层测试

必须覆盖：

- 空会话保存与读取
- 非空队列 round-trip
- 索引与位置 round-trip
- 不完整数据安全降级

### 10.2 service 测试

必须覆盖：

- 启动时成功恢复 queue 与 current index
- 原 `playing` 会话恢复后降级为 `stopped`
- 保存失败不会污染播放状态

### 10.3 集成测试

必须覆盖：

- 启动一个 daemon、构造会话、关闭、重新启动后能看到恢复结果

## 11. 验收标准

本阶段完成时，应满足以下条件：

- 当前播放会话可被持久化到数据库
- daemon 启动后可恢复上一轮 queue、current index 与 position
- 恢复默认是安全的，不会未经确认自动出声
- 保存失败不会破坏播放链路
- 恢复失败会安全降级，不留下脏状态

## 12. 与后续阶段的关系

本阶段完成后：

- 音量与模式可以自然纳入会话持久化
- 后续若加入自动恢复播放，也已有稳定底座
- TUI / Web 未来的“重连即恢复现场”体验将更完整
