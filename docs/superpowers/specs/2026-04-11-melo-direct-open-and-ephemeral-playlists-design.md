# Melo 直接打开入口与临时播放列表设计

日期：2026-04-11

## 1. 目标

为 Melo 补齐“像播放器一样直接打开”的顶层入口，同时不破坏现有 `daemon + remote` 架构。

本设计的核心目标是：

- 让 `melo` 不再表现为“执行了但没有任何效果”
- 让 `melo <音频文件>` 与 `melo <目录>` 成为一等入口，而不是只能通过 `melo play` 遥控已有 queue
- 让直接打开行为沉淀为可管理的“临时播放列表”，而不是一次性运行态 queue
- 让目录入口优先复用库内已知数据，避免每次重新全量读取元数据
- 保持 TUI 始终是统一的交互落点
- 为后续“转正式歌单”“播放历史”“封面增强”等能力保留稳定边界
- 同步补齐一份带完整注释的 `config.example.toml`

一句话概括：

把 `melo` 从“只有遥控命令的 CLI”推进到“可以直接打开文件、目录和当前工作现场的本地播放器入口”。

## 2. 当前现状与问题

当前代码已经具备部分底层能力，但顶层体验还没有收口。

现状中的关键点有：

- `melo` 顶层无参数执行时会直接落到空分支，不产生任何可见行为
- `melo tui` 目前也没有完整接线，用户侧感知接近“没有效果”
- `melo play` 当前只适合作为 daemon 遥控命令，不适合作为“打开文件播放”的入口
- 扫描链路已接入 `Lofty`，可以读取标题、艺术家、专辑、时长、比特率、采样率、位深、声道、歌词和内嵌封面
- 播放与会话恢复链路已经存在，音量、静音、repeat、shuffle 也已有统一快照契约
- 现有媒体库入库按 `songs.path` 唯一键进行 upsert，这为“按路径判断已入库”提供了现成基础
- 现有 `playlists` 只有基础字段，还不能表达“临时”“来源路径”“可见性”“过期时间”等语义

这带来几个直接问题：

- 用户直觉上会期待 `melo xxx.flac` 直接播放，但当前 CLI 信息架构不支持
- 即使底层已有元数据和播放能力，顶层没有统一入口把它们串成可用闭环
- 目录直接打开如果每次都全量重扫，体验会很慢
- 若直接打开只操作内存 queue，后续“保留现场”“转正式歌单”会缺少稳定对象模型

## 3. 备选方向与推荐

### 3.1 方案 A：继续以运行态 queue 作为直接打开的承载对象

做法：

- `melo` / `melo <path>` 只构建 daemon queue
- 不引入新的持久化对象

优点：

- 改动最小
- 实现速度最快

缺点：

- 无法自然支持“隐藏保留”“来源复用”“转正式歌单”
- 单文件和目录入口后续很容易分叉成两套语义
- 不适合承接“最近打开”“过期清理”等后续能力

### 3.2 方案 B：新增独立的临时播放列表表

做法：

- 新建 `temp_playlists` / `temp_playlist_entries`
- 把直接打开入口完全放在独立模型里

优点：

- 语义隔离明显
- 不会直接影响现有 `playlists`

缺点：

- 会复制一套 playlist 能力
- 后续“转正式歌单”需要跨表搬运
- 模型和查询成本更高

### 3.3 方案 C：扩展现有 `playlists`，把临时播放列表建模为 playlist 的一种 kind

做法：

- 在现有 `playlists` 基础上增加来源、可见性和生命周期字段
- 把直接打开生成的对象落为 `kind = ephemeral`

优点：

- 最符合“临时但可管理”的需求
- 自然支持“默认隐藏/显示”“来源复用”“转正式歌单”
- 后续 CLI、TUI、API 都能复用 playlist 体系

缺点：

- 需要扩展现有 playlist 模型和列表过滤规则
- 需要补充 migration

### 3.4 推荐结论

本阶段选择 **方案 C：扩展现有 `playlists`，把临时播放列表建模为 playlist 的一种 kind**。

## 4. 本阶段范围

### 4.1 纳入本阶段

- 顶层 `melo` 默认行为重定义
- 支持 `melo <音频文件>` 和 `melo <目录>` 直接打开
- 引入持久化的临时播放列表模型
- 目录型入口的轻量发现、已入库复用和后台补扫
- 裸 `melo` 的当前目录扫描逻辑
- `melo tui` 的真实落地
- 临时播放列表的来源复用、可见性、TTL 和清理语义
- 临时播放列表转正式歌单的产品边界
- 配置项扩展与 `config.example.toml`
- 相应 CLI/TUI/API/测试约束

### 4.2 不纳入本阶段

- 图片直接打开或独立图片查看
- 播放历史统计与回放界面
- 文件指纹去重或跨路径识别同一首歌
- 云同步
- 多设备会话
- 封面浏览增强交互

图片文件如 `jpg/png/webp` 在本阶段直接返回稳定错误，不作为顶层直接打开格式。

## 5. 顶层入口语义

### 5.1 `melo`

`melo` 不带参数时的默认语义为：

1. 确保 daemon 可用，不可用时自动拉起
2. 若配置允许扫描当前目录，则对当前工作目录做轻量发现
3. 若当前目录命中音频，则将该目录视为一个目录型临时播放列表来源
4. 不论当前目录是否有音频，最终都进入 TUI

这意味着：

- “当前目录有音频”是增强路径，不是进入 TUI 的前置条件
- 即使当前目录没有音频，用户仍可能依赖已有播放列表、库内容或恢复会话

### 5.2 `melo <音频文件>`

执行 `melo <音频文件>` 时：

- 先校验扩展名属于本阶段支持的音频格式
- 以该文件路径为来源创建或刷新一个单项临时播放列表
- 补扫入库后加载到当前 queue
- 自动进入 TUI

单文件入口也统一走“临时播放列表”，而不是绕回仅内存 queue。

### 5.3 `melo <目录>`

执行 `melo <目录>` 时：

- 把目录视为一个目录型临时播放列表来源
- 按配置的递归深度做轻量发现
- 优先复用已入库数据
- 把可播放结果加载到当前 queue
- 自动进入 TUI

### 5.4 不支持格式

本阶段直接打开仅接受音频文件和目录。

若用户执行：

- `melo cover.jpg`
- `melo image.png`
- `melo notes.txt`

则返回稳定错误，例如：

- `unsupported_open_format`

错误文本要明确说明：

- 当前仅支持音频文件和目录
- 图片查看能力将在后续封面功能阶段统一设计

## 6. 支持格式

### 6.1 音频格式

本阶段对齐当前代码已接线能力，直接打开和目录发现支持：

- `flac`
- `mp3`
- `ogg`
- `wav`

是否扩大到更多格式，应在未来先验证 `Lofty` 读取链路与 `rodio` 播放链路的共同可用性，再扩展。

### 6.2 封面与图片格式

本阶段仍保留现有库扫描中的 sidecar / embedded artwork 读取能力，但不提供图片顶层直开。

当前已识别的封面 sidecar 扩展名仍为：

- `jpg`
- `jpeg`
- `png`
- `webp`

## 7. 临时播放列表模型

### 7.1 统一建模

临时播放列表不新建独立表，而是在 `playlists` 上扩展元数据。

建议为 `playlists` 增加：

- `kind`
- `source_kind`
- `source_key`
- `visible`
- `expires_at`
- `last_activated_at`

### 7.2 字段语义

- `kind`
  - `static`
  - `ephemeral`
- `source_kind`
  - `manual`
  - `path_file`
  - `path_dir`
  - `cwd_dir`
- `source_key`
  - 规范化后的绝对路径键
  - 单文件为文件路径
  - 目录为目录路径
- `visible`
  - 是否默认出现在常规播放列表视图中
- `expires_at`
  - `NULL` 表示永不过期
- `last_activated_at`
  - 最近一次被直接打开或激活的时间

### 7.3 单文件也使用临时播放列表

单文件入口统一创建：

- `kind = ephemeral`
- `source_kind = path_file`

原因是：

- 单文件和目录都属于“外部路径来源打开一批可播放项”
- 后续“重新打开”“转正式歌单”“最近打开”“隐藏保留”都需要稳定对象
- 若单文件特殊走 queue，目录走 playlist，后续语义会明显分叉

### 7.4 来源复用

同一路径来源重复打开时，不重复创建新的临时播放列表。

建议查找键为：

- `kind = ephemeral`
- `source_kind`
- `source_key`

命中已有记录时：

- 刷新成员
- 更新 `updated_at`
- 更新 `last_activated_at`
- 重新加载到当前 queue

## 8. 可见性与生命周期

### 8.1 默认可见性

临时播放列表的可见性不写死，而是按来源类型配置默认值。

推荐默认行为：

- `path_file`：默认不展示
- `path_dir`：默认展示
- `cwd_dir`：默认展示

原因是：

- 单文件打开更像一次轻量临时上下文
- 目录打开和当前目录打开更像“目录即歌单”，默认展示更自然

### 8.2 生命周期

临时播放列表默认：

- 持久化到数据库
- 永不过期
- 不自动删除

当配置了 TTL 后：

- 过期对象默认从正常展示面中消失
- 不要求立刻物理删除
- 物理清理由显式命令触发

### 8.3 TTL 语义

推荐约定：

- `expires_at = NULL` 表示永不过期
- 配置层允许设置“默认 TTL 天数”
- 仅当该值大于 0 时，创建临时播放列表时才计算过期时间

### 8.4 清理命令

本阶段应提供显式清理入口，例如：

- `melo playlist cleanup --expired`

该命令负责：

- 删除已过期的临时播放列表
- 级联删除其 entries

默认行为仍是不自动清理。

## 9. 与媒体库的关系

### 9.1 已入库判断

本阶段“是否已入库”按路径判断。

原因是：

- 当前库中 `songs.path` 已是唯一键
- 现有 upsert 逻辑本身就是按路径工作
- 这是当前成本最低且与现有实现最一致的方案

未来若需要支持：

- 文件改名后仍识别为同一首
- 文件移动后保持身份一致

再单独设计文件指纹，不混入本阶段。

### 9.2 统一入库策略

执行 `melo <音频文件>` 或 `melo <目录>` 时：

- 已入库项复用现有库记录
- 未入库项补扫入库
- 最终临时播放列表统一使用库内 `song_id`

本阶段不采用“未入库项只存路径、不入库”的策略。

原因是：

- 后续“转正式歌单”会更自然
- TUI / playlist / queue / search 都能共享同一身份模型
- 避免一半成员有 `song_id`、一半成员只有路径的混合状态

## 10. 目录打开的数据流

### 10.1 轻量发现

目录型入口的第一步不是全量 `Lofty` 扫描，而是轻量发现：

1. 按配置的最大深度遍历目录
2. 收集候选音频路径
3. 对路径做规范化
4. 按路径批量查询库中已有记录

### 10.2 已入库热路径复用

命中库记录后，再结合：

- `path`
- `file_mtime`
- `file_size`

判断是否可直接复用。

若文件未变化，则：

- 不重新读取元数据
- 直接复用已有 `song_id`

这样对“已经入库过的大目录”可以显著减少扫描工作量。

### 10.3 两阶段补扫

为避免拖慢首屏，目录打开采用“两阶段”：

- 第一阶段：优先处理前 `N` 首，形成可播放的最小可用集合
- 第二阶段：其余文件后台继续补扫并回填

其中 `N` 为可配置项，例如：

- `open.prewarm_limit = 20`

### 10.4 后台补扫

后台补扫的要求是：

- 不阻塞进入 TUI
- 不回滚已可播放的上下文
- 若部分文件读取失败，只记录状态和日志，不让整个打开动作失败

## 11. 裸 `melo` 的当前目录行为

### 11.1 默认策略

默认启用“扫描当前目录作为临时播放列表来源”的增强行为。

原因是：

- 用户直接在音乐目录执行 `melo` 时，直觉上会期待“从这里开始”
- 若默认不扫描，用户不容易发现这项能力

### 11.2 扫描范围

当前目录扫描默认采用：

- 递归扫描
- 但带默认最大深度

该深度可配置，例如默认 `2` 或 `3`。

不采用“无限递归且不可控”的默认行为。

### 11.3 无音频时

若当前目录未发现音频：

- 不视为错误
- 不阻止进入 TUI
- 在状态区域提示“当前目录未发现音频”

## 12. TUI 与 CLI 呈现

### 12.1 TUI

TUI 是所有直接打开入口的统一落点。

本阶段要求：

- `melo`
- `melo <file>`
- `melo <dir>`
- `melo tui`

最终都能进入真实可用的 TUI，而不是空实现。

TUI 至少应能表达：

- 当前播放来源
- 当前临时播放列表名称或来源路径
- 当前播放状态
- 连接状态

### 12.2 CLI 顶层结构

顶层命令结构调整原则为：

- `melo play/pause/next/prev/stop/status` 继续保留为遥控命令
- `melo <path>` 成为新的“直接打开”入口
- `melo play` 不再承担“打开文件播放”的语义

帮助文档必须明确区分：

- `melo play` 是遥控 daemon
- `melo <file>` / `melo <dir>` 是直接打开入口

### 12.3 转正式歌单

本阶段应明确保留“临时播放列表转正式歌单”的产品边界。

CLI 可采用类似命令：

- `melo playlist promote <temp-name> <new-name>`

TUI 后续也应能承接同一语义。

## 13. 配置设计

### 13.1 命名原则

配置命名采用“简单项短、复杂项清晰”的混合策略。

规则如下：

- 高频、标量、无需解释的基础项可以短，例如音量
- 复杂行为、容易误解的开关和策略保持较长、较明确的名字
- 不为了“全都很短”牺牲语义清晰度

例如：

- `player.volume`
- `player.restore_last_session`
- `player.resume_after_restore`
- `open.scan_current_dir`

### 13.2 新增配置项

建议新增如下配置域：

```toml
[player]
volume = 100
restore_last_session = true
resume_after_restore = false

[open]
scan_current_dir = true
max_depth = 2
prewarm_limit = 20
background_jobs = 4

[playlists.ephemeral]
default_ttl_days = 0

[playlists.ephemeral.visibility]
path_file = false
path_dir = true
cwd_dir = true
```

其中：

- `player.volume`
  - 默认音量
- `player.restore_last_session`
  - daemon 启动时是否恢复上次播放现场
- `player.resume_after_restore`
  - 恢复后是否自动继续播放
- `open.scan_current_dir`
  - 裸 `melo` 是否尝试扫描当前目录
- `open.max_depth`
  - 当前目录与目录型入口的默认扫描深度
- `open.prewarm_limit`
  - 先保证可播放的预热条目数
- `open.background_jobs`
  - 后台补扫并发数
- `playlists.ephemeral.default_ttl_days`
  - `0` 表示默认永不过期

### 13.3 与现有会话恢复的关系

本阶段不重新发明“记忆上次播放”的机制，而是对齐已有播放会话恢复设计。

要求：

- 恢复现场时保留 queue、当前索引、播放进度
- 恢复现场时保留音量、静音、repeat、shuffle
- 若本次是显式 `melo <file>` / `melo <dir>` 打开，则其优先级高于上次会话
- 仅在没有显式打开路径且当前目录未命中音频时，才回退到“恢复上次现场”

## 14. `config.example.toml`

本阶段必须新增：

- `config.example.toml`

要求如下：

- 覆盖当前项目已有配置项
- 覆盖本阶段新增配置项
- 对每个配置项写中文注释，说明用途、默认语义和典型场景
- 作为用户参考配置存在，不替代开发用 `config.dev.toml`

这意味着 example 中不只包含：

- `database`
- `library.organize`
- `playlists.smart`

也要包含：

- `player`
- `open`
- `playlists.ephemeral`

## 15. 错误处理

### 15.1 不支持格式

返回稳定错误码：

- `unsupported_open_format`

### 15.2 目录无音频

行为：

- 不报致命错误
- 正常进入 TUI
- 给出状态提示

### 15.3 daemon 启动失败

行为：

- CLI 直接退出
- 输出明确错误原因

### 15.4 补扫失败

行为：

- 已可播内容保留
- 失败项记录日志或状态
- 不因为个别文件失败导致整个打开动作失败

## 16. 测试策略

### 16.1 CLI 集成测试

必须覆盖：

- `melo` 裸执行时不再无效果
- `melo` 可自动拉起 daemon 并进入 TUI 路径
- `melo <file>` 对音频有效、对图片报错
- `melo <dir>` 会按深度扫描
- `melo play` 与直接打开入口语义清晰区分

### 16.2 临时播放列表测试

必须覆盖：

- 单文件创建 `path_file` 临时播放列表
- 目录创建 `path_dir` / `cwd_dir` 临时播放列表
- 同一路径重复打开走复用刷新，而不是重复创建
- `visible` 默认值与配置覆盖生效
- 默认 TTL 为永久保留

### 16.3 媒体库复用测试

必须覆盖：

- 已入库且文件未变化时复用现有库记录
- 未入库文件会补扫入库
- 文件变化时会触发重新扫描更新

### 16.4 会话与配置测试

必须覆盖：

- `player.volume` 默认值生效
- `restore_last_session` 与 `resume_after_restore` 语义正确
- 显式路径打开优先级高于恢复现场
- `config.example.toml` 至少通过基本解析校验

## 17. 验收标准

本阶段完成时，应满足以下条件：

- `melo` 不再出现“执行后无任何效果”
- `melo tui` 能进入真实可用的 TUI
- `melo <音频文件>` 可以直接进入 TUI 并开始播放
- `melo <目录>` 可以创建目录型临时播放列表并进入 TUI
- 临时播放列表落库并可按来源复用
- 单文件与目录入口统一建立在 `ephemeral playlist` 上
- 当前目录默认可作为临时播放列表来源，但无音频时也不会阻止进入 TUI
- 目录入口优先复用已入库数据，而不是每次全量重扫
- 默认不过期，仅在配置 TTL 后才参与过期逻辑
- 仓库中存在一份带完整中文注释的 `config.example.toml`

## 18. 后续扩展点

本设计完成后，可以自然继续推进：

- 最近打开列表

- 基于文件指纹的跨路径身份识别

其中播放历史建议独立建模，不与临时播放列表混在同一表中。
