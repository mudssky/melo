# Melo 数据库自动初始化设计

日期：2026-04-12

## 背景

当前 `melo <音频文件>` 这类 direct-open 入口会先尝试自动拉起 daemon，再通过 API 执行打开逻辑。实际运行中，如果配置里的数据库路径指向一个尚未存在父目录的位置，例如 `local/melo-dev.db`，SQLite 会在连接阶段直接报：

```text
unable to open database file
```

用户最终看到的是 API 包装后的数据库连接错误，而不是一个明确的“数据库尚未初始化/目录不存在”的运行时准备失败提示。

项目中已经存在数据库 bootstrap 逻辑，能够负责创建父目录并执行迁移；问题在于这套逻辑没有被统一纳入 daemon 启动链路，导致业务请求先于数据库准备发生。

## 目标

- 在 daemon 启动阶段统一完成数据库运行时准备
- 首次使用 `melo <文件>` 时无需手动执行 `melo db init`
- 当数据库父目录不存在时自动创建并初始化 schema
- 当数据库准备失败时，错误应稳定暴露为“启动准备失败”，而不是业务 API 半路报数据库连接错误
- 保持职责清晰，避免把初始化副作用下沉到底层 `connect()` 或分散到各个业务服务中

## 非目标

- 不重做整套数据库命令语义
- 不把数据库准备逻辑塞进每个 repository 或 API handler
- 不在本次设计中扩展新的数据库恢复、修复或备份能力
- 不修改只读且不触库的 CLI 命令行为

## 方案概览

新增统一的“数据库运行时准备”入口，并在 daemon 启动前强制执行一次。direct-open 等依赖自动拉起 daemon 的入口不单独处理，而是复用 daemon 启动链路带来的数据库准备能力。

数据库准备入口负责：

1. 解析最终数据库路径
2. 确保数据库父目录存在
3. 建立数据库连接
4. 执行迁移，确保 schema 可用

如果任一步骤失败，则 daemon 启动失败，并返回明确的 Melo 级错误。只有数据库准备成功后，daemon 才继续构造应用状态并对外提供 API。

## 方案对比

### 方案 A：只在 direct-open 入口前补初始化

优点：

- 改动最小
- 可以快速覆盖 `melo <文件>` 这条当前已知失败路径

缺点：

- `melo daemon start` 与其他未来触库入口仍可能重复踩坑
- 初始化责任散落在 CLI 特定入口，不利于维护
- 只能修症状，不能修“daemon 启动前未统一准备运行环境”的根因

### 方案 B：在 daemon 启动阶段统一执行数据库准备

优点：

- 职责边界最清晰
- direct-open、TUI、daemon API 统一受益
- 错误会稳定收敛到启动阶段，便于理解与排查
- 可以复用现有 bootstrap 实现，不需要把副作用塞到连接层

缺点：

- 需要调整 daemon 启动路径和相关测试

### 方案 C：把初始化下沉到数据库连接函数中

优点：

- 最自动，调用方不用关心

缺点：

- 副作用过深，连接层职责不再单一
- 连接失败和初始化失败会混在一起，调试更绕
- 后续如果要区分“单纯连库”和“执行迁移”会更麻烦

## 结论

采用方案 B：在 daemon 启动阶段统一执行数据库准备。

这是当前最稳的边界划分。数据库准备属于“运行时环境就绪”的一部分，应在应用状态和业务服务构造前完成，而不是分散在业务逻辑中临时触发。

## 模块设计

### `src/core/db/bootstrap.rs`

作为数据库准备的核心模块，保留它对以下职责的统一封装：

- 创建数据库父目录
- 建立数据库连接
- 执行迁移

如果当前接口表达不够直接，可以补一个更清晰的公开入口，例如 `prepare_runtime_database`，但不改变其核心职责。

### `src/daemon/app.rs`

`AppState::new()` 在创建业务服务前先调用数据库准备入口。

顺序要求如下：

1. 加载 `Settings`
2. 执行数据库准备
3. 构造播放后端、library service、playlist service、open service 等运行时组件

这样 daemon 一旦启动成功，就意味着数据库已经处于可用状态。

### `src/cli/run.rs`

不为 direct-open 增加额外的数据库初始化逻辑。

原因：

- `melo <文件>` 已经依赖自动拉起 daemon
- 如果 CLI 和 daemon 两边都做初始化，会造成职责重复
- 统一复用 daemon 启动前准备，链路更简单

### 错误映射层

数据库准备失败时，应优先返回更明确的 Melo 级错误，例如：

- `failed to prepare database`
- `cannot create database directory: ...`
- `database migration failed: ...`

目标不是完全隐藏底层原因，而是把用户首先看到的错误语义从“业务 API 调用失败”改成“本地运行环境未准备完成”。

## 数据流

### 成功路径

1. 用户执行 `melo '.\Always Online - 林俊杰.flac'`
2. CLI 进入 direct-open 分支
3. CLI 调用 `ensure_running`
4. daemon 启动
5. daemon 启动前执行数据库准备
6. 若数据库目录不存在，则自动创建；若数据库文件不存在，则自动初始化并迁移
7. daemon 启动成功
8. CLI 调用 daemon API 执行 `open_target`
9. 业务逻辑正常访问数据库并继续播放流程

### 失败路径

1. 用户执行触库入口
2. daemon 启动前执行数据库准备
3. 父目录创建失败或迁移失败
4. daemon 启动直接失败
5. CLI 输出数据库准备失败的明确错误
6. 不进入后续业务 API 调用

## 测试策略

### 必要测试

1. daemon 启动时，如果数据库路径的父目录不存在，应自动创建目录并成功完成初始化
2. direct-open CLI 集成测试中，在数据库目录不存在的前提下应仍能成功走到启动链路

### 推荐补充测试

1. 当数据库父目录不可创建时，应返回包含 `failed to prepare database` 的错误
2. 已存在数据库时再次启动 daemon 不应破坏现有流程

## 风险与取舍

### 风险

- 如果把准备逻辑挂在错误的层级，可能让测试环境或只读命令也被迫触发数据库副作用
- 如果错误映射过度包装，可能丢失底层诊断信息

### 取舍

- 本次优先保证首次使用体验和职责清晰
- 只读且不触库的命令不主动初始化数据库
- 错误文案以“先说明阶段，再附具体原因”为准，兼顾可读性与可诊断性

## 验收标准

- 在数据库父目录不存在的情况下，`melo <音频文件>` 不再因为 `unable to open database file` 失败
- daemon 启动成功意味着数据库目录、文件和 schema 都已处于可用状态
- 数据库准备失败时，CLI 不再只显示底层 API 错误，而能指出启动阶段的数据库准备失败
- 新增测试能够稳定覆盖首次启动自动初始化场景
