# Melo `mpv` 客户端模式与 `libmpv` 演进设计

日期：2026-04-13

## 1. 背景

当前 Melo 已经具备：

- daemon 持有播放器状态与队列真相源
- TUI 作为前台交互界面
- `mpv` 作为可用播放后端

但在产品体验上，`mpv` 集成方式仍然不够像一个“终端客户端”，而更像一个“遥控外部播放器”的方案，问题集中在以下几点：

- 默认播放时可能出现独立 `mpv` 窗口
  - 它会抢占焦点、影响 TUI 操作
  - 任务栏额外出现一个 `mpv` 也破坏产品统一感
- TUI 退出时当前播放不会默认停止
  - 用户体感上更像“控制面退出了，但播放器还在外面自己播”
  - 这不符合期望中的 `nvim` 式客户端语义
- `mpv` 被手动关闭后，当前实现容易表现成“切歌”
  - 这是因为播放结束原因没有被正确细分
  - “自然播完”和“用户主动关闭后端”不应该走同一套逻辑
- 当前播放信息没有在 TUI 内闭环
  - 右侧歌单预览没有高亮当前播放曲目
  - 歌词与封面没有成为 TUI 正式信息架构的一部分

用户期望的方向是：

- Melo TUI 更像一个真正的终端音乐客户端
- `mpv` 应该更像后台播放引擎，而不是会跳出来干扰的外置 GUI
- 默认体验不弹独立窗口
- 歌词、封面、当前播放曲目等信息应尽量回到 TUI 内显示

## 2. 目标

本次设计目标如下：

- 把 Melo 的产品定位明确为：
  - `TUI 是主客户端`
  - `mpv` 是后台播放引擎
- 让当前 `mpv-ipc` 后端进入真正的 headless client mode
- 修正播放生命周期语义：
  - 退出 TUI 时默认停止当前播放
  - 手动关闭 `mpv` 时解释为 `stop`
  - 只有自然 EOF 才允许触发切歌/循环逻辑
- 补齐 TUI 内的播放信息闭环：
  - 当前播放曲目高亮
  - 歌词正式显示
  - 封面区进入正式布局并支持能力降级
- 为未来 `libmpv` 后端预留清晰抽象，但不在本期接入

## 3. 非目标

本次明确不做：

- 一期不接入 `libmpv`
- 不再引入 `GStreamer` 或 `FFmpeg` 作为主播放后端候选
- 不把外部 `mpv` 窗口模式作为默认体验
- 不在一期保证所有终端都能显示真实封面图像
- 不在一期大幅重做整个 TUI 多视图结构
- 不把视频播放或复杂媒体可视化纳入范围

## 4. 方案结论

本次采用两期方案：

### 4.1 一期

继续使用当前 `mpv-ipc` 路线，但把它升级为正式的 client mode：

- 默认无独立窗口
- 默认不抢焦点
- TUI 退出默认停播
- `mpv` 手动关闭默认视为 `stop`
- 当前播放曲目、歌词、封面信息回到 TUI 内部闭环

### 4.2 二期

新增 `libmpv` 后端作为可选路线：

- 复用一期确定下来的生命周期与 UI 语义
- 对比 `mpv-ipc` 与 `libmpv` 的稳定性、部署复杂度和产品手感
- 再决定未来是否切换默认后端

## 5. 为什么只保留 `mpv-ipc` 与 `libmpv`

### 5.1 不保留 `GStreamer`

`GStreamer` 是成熟媒体框架，但对 Melo 这个场景来说更像一条新媒体内核路线，而不是当前 `mpv` 方案的演进版。

它的问题不在能力，而在：

- 会显著改变工程心智模型
- 增加额外 runtime / plugin 体系
- 与当前 `mpv` 用户预期和配置链路距离更远

因此不作为当前设计主线。

### 5.2 不把 `FFmpeg` 作为主播放后端

`FFmpeg` 更适合作为底层媒体处理工具箱，而不是现成播放器后端。

如果直接用它做主播放后端，应用侧需要自己补齐：

- 音频输出
- 播放时钟
- 暂停/恢复/停止语义
- EOF / stop / 异常退出事件
- 更完整的状态同步

这与本次目标“让产品更像客户端”不匹配，因此不纳入本轮主候选。

## 6. 产品定位

本次设计明确将 Melo 的产品语义定义为：

- Melo TUI 是主客户端，不是遥控器
- 后台播放引擎是可替换实现
- 用户应主要停留在 TUI 中完成浏览、播放、看歌词、看封面等动作

这意味着：

- 默认播放不应弹独立 GUI 播放器窗口
- 外部窗口模式最多只是后续可配置模式
- 当前播放信息不能依赖外部窗口补足

## 7. 生命周期语义

### 7.1 退出 TUI

默认行为固定为：

- `退出 TUI = 停止当前播放，但 daemon 继续存活`

解释：

- 声音应停止
- 当前播放引擎实例应显式 stop / teardown
- daemon 保留，方便下次秒开与状态复用

后续可以扩展配置项，但一期默认语义不变。

### 7.2 手动关闭 `mpv`

当用户手动关闭 `mpv` 时，Melo 默认应解释为：

- `UserClosedBackend`
- 其产品语义等同于 `UserStop`

它不应触发：

- 自动切下一首
- 自动循环
- 自动继续排队

### 7.3 自然播完

只有当当前曲目自然 EOF 时，才允许触发：

- 下一首
- repeat
- shuffle

因此，播放结束原因不能再被简化成单一的“曲目结束”。

### 7.4 内部结束原因模型

一期建议把后端结束原因明确拆成至少以下几类：

- `NaturalEof`
- `UserStop`
- `UserClosedBackend`
- `BackendAborted`

规则：

- `NaturalEof`：进入正常切歌 / repeat / shuffle 逻辑
- `UserStop`：停在当前上下文，不自动切歌
- `UserClosedBackend`：视同 `UserStop`
- `BackendAborted`：进入错误态或至少给出明确提示，不自动切歌

## 8. 一期 `mpv-ipc` 设计

### 8.1 默认运行模式

一期中，`mpv-ipc` 应作为正式后端继续存在，但默认运行模式必须明确进入 headless client mode：

- 默认不弹独立窗口
- 默认不抢占焦点
- 默认不额外污染任务栏心智
- 从产品角度看，更像“无界面的后台播放内核”

### 8.2 配置语义

当前配置不应继续把 `mpv` 视为单一实现。

为了给二期铺路，建议在设计层面区分：

- `mpv_ipc`
- `mpv_lib`（保留位，二期实现）

一期实际只实现 `mpv_ipc`，但配置、命名、抽象层避免把 `mpv` 写死成唯一后端实现。

### 8.3 事件解析

一期必须修复当前 `mpv-ipc` 的事件语义：

- 不能把所有 `end-file` 都等价映射为“自然播完”
- 必须能够区分：
  - 自然 EOF
  - 主动 stop
  - 用户关闭后端
  - 后端异常退出

如果 `mpv-ipc` 无法直接提供完整语义，就应在 Melo 一侧引入状态机补足区分。

## 9. TUI 信息架构

### 9.1 总原则

TUI 必须形成播放信息闭环，不能依赖外部 `mpv` 窗口补足状态。

### 9.2 左侧列表

左侧仍保留 playlist-first 结构，但继续明确区分：

- 当前浏览选择
- 当前实际播放来源

### 9.3 右侧区域

右侧区域升级为围绕“当前播放”的详情面板，而不是单纯的歌单预览。

包含两层信息：

- 当前播放上下文
  - 当前曲目
  - 当前播放来源
  - 当前运行目录
  - 播放状态 / repeat / shuffle
- 内容区
  - 当前歌单曲目列表
  - 当前播放曲目高亮
  - 歌词
  - 封面

### 9.4 当前播放曲目高亮

当右侧当前显示的曲目列表对应当前播放来源时：

- 必须高亮当前播放曲目
- 该高亮与“当前选择行”是不同语义
- 允许：
  - 当前选择行
  - 当前播放行
  - 二者重合时的组合态

### 9.5 歌词

歌词在一期成为正式 TUI 组件：

- 文本歌词优先
- 无歌词时有稳定占位
- 后续可扩展多语言/多歌词源切换

### 9.6 封面

封面在一期进入正式布局，但遵循终端能力分级：

- 支持终端图片协议时显示真实封面
- 不支持时回退为稳定占位卡片 / 文本元数据面板

封面是增强体验，不应成为核心播放体验依赖。

## 10. 二期 `libmpv` 设计预留

二期引入 `libmpv` 时，不重新定义产品语义，只更换播放内核实现。

也就是说，二期只做：

- 新增 `libmpv backend`
- 复用一期已经定好的：
  - 生命周期语义
  - 结束原因模型
  - TUI 信息架构
  - 当前播放曲目映射逻辑

这样可以保证：

- 一期与二期对用户来说是同一个产品
- 只是底层播放实现不同

## 11. 跨平台结论

### 11.1 一期 `mpv-ipc`

在有可用音频输出环境时，`mpv-ipc` headless 模式可以覆盖：

- Windows
- macOS
- Linux 桌面环境

对于纯 Linux server：

- 如果不存在可用音频设备或音频服务
- 则问题不在 TUI，而在本机没有可播放输出环境

### 11.2 封面显示

封面显示能力不等于播放能力。

图片封面属于终端增强能力，依赖终端协议支持，因此需要能力降级：

- Kitty / iTerm2 / Sixel / 类 ueberzug 协议优先
- 其他终端回退为文本/占位 UI

### 11.3 二期 `libmpv`

`libmpv` 理论上也可跨平台，但原生依赖与分发复杂度高于 `mpv-ipc`。

因此它更适合作为二期后端，而不是一期直接切换的默认方案。

## 12. 参考实现启发

`termusic` 的实现说明了以下方向是可行的：

- 使用 `libmpv` 做真正的无窗口播放后端
- 在 TUI 内独立显示歌词
- 在 TUI 内通过终端协议显示封面
- 把“退出 TUI 是否顺带退出 server”作为清晰行为选项

本设计借鉴它的产品形态，但不在一期直接照搬其 `libmpv` 路线，而是采用：

- 一期先修正 `mpv-ipc`
- 二期再引入 `libmpv`

## 13. 一期范围

一期包含：

- `mpv-ipc` 默认无窗口 client mode
- TUI 退出默认停播
- 手动关闭 `mpv` 等价 `stop`
- 自然 EOF 与 stop reason 分离
- 当前播放曲目高亮
- 歌词正式显示
- 封面正式进入布局并支持能力降级
- 配置与抽象层预留 `mpv_lib`

一期不包含：

- 真实接入 `libmpv`
- 替换默认播放后端
- 引入第三条主播放后端路线

## 14. 二期范围

二期包含：

- 新增 `libmpv` 后端
- 支持在 `mpv_ipc` 与 `mpv_lib` 间切换
- 对比两者在稳定性、部署复杂度、行为一致性上的表现

二期不包含：

- 重做一期 UI 信息架构
- 重定义播放生命周期语义

## 15. 验收标准

### 15.1 一期

- 默认播放不再弹独立 `mpv` 窗口
- TUI 操作过程中不会被 `mpv` 抢焦点
- 退出 TUI 后声音停止
- 退出 TUI 后 daemon 仍可复用
- 手动关闭 `mpv` 不会触发“切下一首”
- 当前播放曲目在 TUI 内可见且可高亮
- 歌词在 TUI 内可见
- 封面支持终端协议时显示，不支持时优雅降级

### 15.2 二期

- `libmpv` 可作为可选后端启动
- 与一期定义的客户端语义一致
- 默认无窗口体验成立
- 上层歌词 / 封面 / 当前播放信息逻辑不因后端切换而改变

## 16. 结论

本次设计不试图在一期一步到位替换播放内核，而是先修正产品语义：

- 把 Melo 从“遥控外部播放器”收敛为“主客户端 + 后台播放引擎”
- 让当前 `mpv-ipc` 进入真正的 headless client mode
- 把当前播放曲目、歌词、封面信息收回到 TUI 内闭环
- 在此基础上，把 `libmpv` 留作更贴近最终产品形态的二期后端

这样可以在控制风险的前提下，先解决当前最影响体验的违和感，再为后续演进保留清晰路径。
