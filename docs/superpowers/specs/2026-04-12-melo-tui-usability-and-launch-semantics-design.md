# Melo TUI 可用性与启动语义整合设计

日期：2026-04-12

## 1. 背景

当前 Melo 的 TUI 已经具备基础的数据链路，但离“可投入使用”还有明显差距，问题集中在四组体验断层上：

- 裸 `melo` / `melo --verbose` 的默认语义与用户直觉不一致
  - 当前开发包装器会把工作目录固定到仓库根目录
  - 导致“当前目录”被错误解释成项目目录，而不是用户实际运行命令的目录
- 默认启动语义过于单一
  - 当前裸 `melo` 倾向于直接对当前目录做 `cwd_dir` direct-open
  - 但当 daemon 已经在播放有声音的会话时，用户期望是继续播放，而不是强行切换来源
- TUI 交互几乎不可感知
  - 目前焦点切换、选中状态、当前播放来源等反馈主要依赖文本字符，缺少真正的视觉高亮
  - 鼠标尚未接入正式语义
  - 键盘交互虽有雏形，但还不足以支撑高频使用
- `--verbose` 与 TUI 的边界不清晰
  - verbose 日志在启动链路中有排障价值
  - 但一旦进入 TUI，继续让终端输出与界面争抢注意力，会破坏使用体验

这次设计的目标不是单独补一个 bug，也不是简单给现有列表加几行样式，而是把 Melo 的默认启动语义、TUI 首页交互模型、键位系统和 verbose 边界一起整合成一个可投入使用的最小完整闭环。

## 2. 目标

本次设计目标如下：

- 修正开发包装器，让 `melo` 在开发态保留调用时工作目录，而不是强制切到仓库根目录
- 明确裸 `melo` / `melo --verbose` 的默认启动语义
  - 有活动播放时继续播放
  - 无活动播放时切换到当前运行目录
- 保持显式路径优先级不变
  - `melo D:/Music`
  - `melo ./song.flac`
- 把 Playlist 首页补成一个真正可操作的界面
  - 焦点可见
  - 选中可见
  - 当前播放来源可见
  - 鼠标单击/双击语义明确
  - 键盘交互完整可用
- 引入一个正式的、可配置的通用 keymap 系统
  - 支持单键
  - 支持组合键
  - 支持键序列
  - 支持 Vim 风格别名
- 让 `--verbose` 与 TUI 的边界变得合理
  - 进入 TUI 前可以输出启动日志
  - 进入 TUI 后终端完全交给 TUI

## 3. 非目标

本次明确不做：

- 不把 TUI 首页重构成新的多视图工作台
- 不新增日志面板、日志抽屉或内嵌日志窗口
- 不加入右键菜单、拖拽排序、上下文菜单
- 不在 TUI 内实现 playlist 管理操作，例如重命名、删除、提升 ephemeral playlist
- 不把 `Songs / Search / Queue` 多视图一起做完
- 不把 keymap 扩展成完整的模式化编辑器系统
- 不在本次引入任意命令字符串绑定或脚本化宏系统

## 4. 方案结论

本次采用“交互基础层”方案：

- 启动语义统一修正
- TUI 首页维持当前 playlist 为中心的信息架构
- 渲染层从纯文本拼接升级为状态化交互组件
- 输入层引入统一的 `input -> intent -> effect` 模型
- 键盘与鼠标复用同一套动作语义
- keymap 改为配置驱动

相比“只修 bug”的补丁方案，这个方向能一次性解决：

- 当前目录被错误解释成项目目录
- 裸 `melo` 抢占已有播放会话
- 焦点切换与选中几乎不可感知
- 鼠标不可用
- 快捷键不可配置
- `--verbose` 与 TUI 相互干扰

同时，它又明显小于一次全量 TUI 重构，适合作为“可投入使用”的第一版。

## 5. 启动语义设计

### 5.1 开发包装器工作目录

开发包装器继续负责：

- 注入 `config.dev.toml`
- 调起 cargo install 后的 `melo.exe`

但不再把子进程工作目录硬编码为仓库根目录。

设计要求：

- 包装器默认继承调用方 shell 的当前工作目录
- 开发态与正式二进制在“当前目录”语义上保持一致
- `MELO_CONFIG` 注入逻辑保留，不与工作目录耦合

这样可以保证：

- 用户在 `D:/Music/Aimer` 中执行 `melo`
- TUI 启动语义看到的“当前目录”
- 与用户 shell 里实际所在目录一致

### 5.2 裸 `melo` / `melo --verbose` 的默认决策

裸启动不再一律执行 `cwd_dir` direct-open，而是先做一次“启动落点决策”。

决策结果固定为三类：

- `PreserveCurrentSession`
- `OpenLaunchCwd`
- `OpenExplicitTarget`

规则如下：

1. 如果用户显式传入路径，则总是走 `OpenExplicitTarget`
2. 如果用户没有显式路径：
   - 当播放器处于实际播放中，并且存在可识别的当前播放来源时，走 `PreserveCurrentSession`
   - 当当前没有活动播放时，走 `OpenLaunchCwd`

这里“有活动播放”固定解释为：

- 当前播放器状态为 `Playing`
- 用户仍能听到声音的播放会话

这意味着：

- `Paused`、`Stopped`、空会话，不视为“继续保留”
- 只有正在播放的会话，才阻止当前运行目录自动接管

### 5.3 启动落点对 TUI 的影响

三个落点的默认行为如下：

- `PreserveCurrentSession`
  - 不执行新的 direct-open
  - 保持当前播放继续
  - TUI 左栏默认高亮当前播放来源
  - 右上明确展示“当前运行目录”作为次上下文
- `OpenLaunchCwd`
  - 对当前运行目录执行 `cwd_dir` direct-open
  - 左栏默认高亮当前运行目录对应的 ephemeral playlist
  - 当前播放来源同步变成该目录来源
- `OpenExplicitTarget`
  - 沿用现有显式路径 direct-open 语义
  - 左栏默认高亮显式目标对应 playlist

## 6. 状态模型

### 6.1 daemon 全局状态

daemon 继续作为全局真相源，持有：

- 当前播放器快照
- 当前播放来源 playlist
- playlist 列表聚合快照

daemon 不维护 TUI 局部浏览焦点，也不维护键位上下文。

### 6.2 TUI 本地状态

TUI 本地状态负责：

- 当前焦点区域
- 当前高亮 playlist
- 当前高亮歌曲
- 当前鼠标 hover 区域
- 当前按键序列缓冲
- 当前帮助弹层开关
- 当前预览缓存与滚动位置
- 当前启动上下文
  - `launch_cwd`
  - `startup_notice`

### 6.3 播放来源与浏览选择分层

本次设计继续明确区分：

- 当前播放来源
- 当前浏览高亮

二者默认可以一致，但不要求永远一致。

这样可以同时满足：

- 当前声音继续播放
- 用户进入 TUI 后继续浏览别的 playlist
- 浏览选择变化不打断播放

## 7. 交互模型

### 7.1 输入分层

TUI 输入统一拆成三层：

- `Input`
  - 键盘事件
  - 鼠标事件
- `Intent`
  - 抽象动作，例如：
    - `MovePlaylistUp`
    - `MovePlaylistDown`
    - `FocusNextPane`
    - `PlaySelection`
    - `ToggleHelp`
- `Effect`
  - 仅更新本地状态
  - 或触发远端 API 调用

这个分层的目的，是让键盘和鼠标共享同一套动作定义，避免交互语义在不同输入设备之间漂移。

### 7.2 焦点区域

Playlist 首页第一版固定两个主焦点区：

- `PlaylistList`
- `PlaylistPreview`

默认规则：

- 启动后默认焦点在左侧 `PlaylistList`
- `Tab` / `Shift+Tab` 在两区之间循环
- 焦点切换只改变操作区域，不自动播放
- 左侧选中变化只刷新右侧预览，不自动播放

### 7.3 鼠标语义

第一版鼠标语义固定如下：

- 左键单击
  - 切换焦点到点击区域
  - 更新高亮项
  - 若点击左侧 playlist，则刷新右侧预览
- 左键双击
  - 执行与 `Enter` 完全一致的播放动作
- 鼠标滚轮
  - 在 hover 区域内滚动
  - 若没有 hover 区域，则作用于当前焦点区域

第一版不做：

- 右键菜单
- 拖拽
- 三击、多击语义

## 8. 键盘交互与 keymap

### 8.1 最低可用键盘集

第一版按“可投入使用”标准提供完整键盘集：

- 焦点
  - `Tab`
  - `Shift+Tab`
  - `Esc`
- 导航
  - `Up` / `Down`
  - `Home` / `End`
  - `PageUp` / `PageDown`
- 播放确认
  - `Enter`
- 播放器控制
  - `Space`
  - `<`
  - `>`
  - `r`
  - `s`
- 帮助与退出
  - `?`
  - `q`
- Vim 风格别名
  - `j` / `k`
  - `h` / `l`
  - `gg`
  - `G`

固定语义：

- `Enter`：播放当前选择
- `Esc`：优先关闭帮助/弹层，其次回到左侧列表焦点
- `h / l`：切换焦点区，而不是切视图
- `gg / G`：跳到列表头尾

### 8.2 keymap 配置模型

新增 `tui.keymap` 配置区，采用“动作驱动”结构，而不是“按键驱动”结构。

示例：

```toml
[tui.keymap]
focus_next = ["tab", "l"]
focus_prev = ["shift+tab", "h"]
move_up = ["up", "k"]
move_down = ["down", "j"]
jump_top = ["home", ["g", "g"]]
jump_bottom = ["end", "shift+g"]
page_up = ["pageup", "ctrl+b"]
page_down = ["pagedown", "ctrl+f"]
play_selection = ["enter"]
toggle_playback = ["space"]
toggle_repeat = ["r"]
toggle_shuffle = ["s"]
open_help = ["?"]
quit = ["q"]
```

语法约定：

- 单键：`"enter"`
- 组合键：`"shift+tab"`
- 键序列：`["g", "g"]`

### 8.3 动作 ID

每个可配置动作都有稳定 action id，例如：

- `focus_next`
- `focus_prev`
- `move_up`
- `move_down`
- `jump_top`
- `jump_bottom`
- `page_up`
- `page_down`
- `play_selection`
- `toggle_playback`
- `toggle_repeat`
- `toggle_shuffle`
- `open_help`
- `quit`

帮助弹层、底部提示、测试断言都基于这些 action id，而不是基于硬编码文本。

### 8.4 KeymapResolver

新增 `KeymapResolver` 负责：

- 解析当前按键是否命中完整动作
- 识别当前输入是否是一个合法前缀
- 在键序列失效或超时时回退
- 输出统一的 `Intent`

第一版保持全局单一 keymap，不引入模式切换。

## 9. 视觉反馈与渲染

### 9.1 视觉状态

这次不能再只依赖 `>` 作为交互反馈，必须正式支持三类视觉状态：

- `focused`
- `selected`
- `currently_playing_source`

同一个列表项可以同时拥有多种语义，例如：

- 它既是当前播放来源
- 又是当前高亮项

因此渲染层不能再只依赖单一布尔值。

### 9.2 渲染要求

首页布局仍保持四区：

- 左栏：播放列表
- 右上：当前播放来源 / 当前运行目录 / 模式
- 右下：歌单预览
- 底栏：Now Playing / 状态 / 快捷键提示

渲染层升级要求：

- 左栏有明确焦点边框
- 当前选中项有背景或前景反差
- 当前播放来源有额外标识
- 右下预览支持独立焦点与高亮
- 帮助文案从 keymap 解析结果动态生成，不再写死

## 10. verbose 与 TUI 的边界

`--verbose` 的行为固定为：

- 进入 TUI 前：
  - 可以输出 CLI startup 阶段日志
  - 可以输出 daemon 启动日志镜像
- 成功进入 TUI 后：
  - 停止终端日志镜像
  - 终端完全交给 TUI
- 启动失败时：
  - 终端保留最近关键日志，便于排障

本次不做：

- TUI 内嵌日志面板
- 运行中持续日志流

## 11. 测试策略

### 11.1 启动语义测试

覆盖：

- dev wrapper 不再篡改调用时工作目录
- 裸 `melo` 在 `Playing` 状态下保留当前播放会话
- 裸 `melo` 在无活动播放时 direct-open 当前运行目录
- 显式路径优先级高于默认语义

### 11.2 交互层测试

覆盖：

- `Tab` / `Shift+Tab`
- `Esc`
- `Home` / `End`
- `PageUp` / `PageDown`
- `j/k`
- `h/l`
- `gg`
- `G`
- 鼠标单击选中
- 鼠标双击播放

### 11.3 keymap 测试

覆盖：

- 单键解析
- 组合键解析
- 键序列解析
- 前缀等待与超时回退
- 覆盖默认 keymap 的配置行为
- 帮助文案和 footer hints 是否读取解析后的 keymap

### 11.4 TUI 状态测试

覆盖：

- 当前播放来源与当前浏览高亮的分层语义
- 高亮变化只刷新预览，不自动播放
- `Enter` 与双击是否触发同一动作
- 当前运行目录是否作为次上下文进入右上状态区

### 11.5 verbose 边界测试

覆盖：

- `--verbose` 在进入 TUI 前能输出启动日志
- TUI 启动成功后终端不再继续刷 verbose 日志
- 启动失败时能保留关键排障日志

## 12. 风险与取舍

### 风险

- 如果 keymap 设计得过于随意，后续动作扩展会很快变乱
- 如果把当前播放来源和当前浏览高亮重新耦合，仍然会出现“浏览一下就打断播放”的问题
- 如果只补样式不补状态模型，高亮效果会继续流于表面

### 取舍

- 本次优先建立“可用的交互基础层”，而不是做更大的首页改版
- 本次只做单一全局 keymap，不做模式系统
- 本次让 verbose 在进入 TUI 后退场，而不是把日志也纳入首页设计

## 13. 验收标准

- 开发态 `melo` 的“当前目录”语义与用户 shell 当前目录一致，不再固定为仓库根目录
- 裸 `melo` 在已有活动播放时保持当前声音继续播放
- 裸 `melo` 在无活动播放时切换到当前运行目录
- 左栏和右下区域拥有明确的焦点与高亮反馈
- 鼠标单击可选中，双击可播放
- 键盘交互达到可投入使用标准
- 快捷键通过配置驱动，可绑定单键、组合键和键序列
- 帮助文案与底部提示不再硬编码，而是反映有效 keymap
- `--verbose` 不再在 TUI 运行中污染终端展示

## 14. 结论

本次设计把 Melo 当前零散的几个问题收敛成一个统一的“可用性整合”目标：

- 修正默认启动语义
- 保留已有播放会话的连续性
- 让 Playlist 首页具备真正可感知的交互
- 建立通用且可配置的 keymap 基础
- 把 verbose 与 TUI 的边界重新划清

完成这一轮后，Melo 的 TUI 才能从“能显示数据”升级到“能稳定使用”。后续无论继续扩展 `Songs / Search / Queue`，还是加入更复杂的输入模式、命令面板、日志面板，都可以建立在这套基础之上。
