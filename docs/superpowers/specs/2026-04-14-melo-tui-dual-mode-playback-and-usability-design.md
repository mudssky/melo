# Melo TUI 双模式播放与可用性整合设计

日期：2026-04-14

## 1. 背景

当前 Melo 的 TUI 与 daemon 播放链路已经具备“能工作”的基础，但离高频可用还有明显距离，问题集中在以下几组体验断层：

- 切歌、切模式、播放控制时体感卡顿
  - 当前 TUI 在主循环里同步等待部分远端请求
  - 当前 daemon 会把高频播放状态变化放大成整页聚合快照重建
- 底部播放信息不会顺滑地随播放更新
  - 当前播放进度是高频状态
  - 但 TUI 消费的是偏重的聚合模型
- 列表可用性不足
  - 选中项走出可视区时，视口不会自动跟随
  - 长列表没有统一滚动条模型
  - 长标题没有统一裁剪策略
- 歌词区几乎没有形成“播放器歌词体验”
  - 当前歌词不会随播放自动滚动
  - 当前播放歌词行没有高亮
  - 用户无法手动滚动查看之前的歌词后再平滑回到当前句
- 当前 TUI 设计与未来前端模型边界不清
  - 未来 Web 前端的主形态应是“前端本地播放”
  - daemon 播放只是少用模式
  - 因此不能把现有 daemon 播放协议直接当成所有前端的长期母模型

本次设计的目标不是只修一个滚动 bug，也不是只给当前 TUI 打补丁，而是把下面四件事一起收敛：

- daemon 播放模式的高频交互体验
- TUI 首页的信息架构与滚动/歌词体验
- 面向未来 Web 前端的数据层/播放层边界
- 共享播放模式与内容模型

## 2. 目标

本次设计目标如下：

- 让 TUI 在 daemon 播放模式下具备稳定、顺滑的播放控制体验
  - `next`
  - `prev`
  - `toggle`
  - 模式切换
- 把高频播放状态同步从重聚合快照里拆出来，形成轻量运行时状态流
- 让 TUI 引入正式的本地缓存与 UI 状态模型
  - 列表 viewport
  - 滚动条
  - 歌词跟随状态
  - 乐观交互状态
- 重做 TUI 首页为“双中心”结构
  - 浏览内容与当前播放同等重要
  - 布局接近对半
- 让所有长列表具备统一的滚动、裁剪和滚动条行为
- 让歌词区支持完整播放器体验
  - 当前行高亮
  - 自动跟随播放
  - 手动回看历史歌词
  - 手动滚动后默认 `3s` 恢复自动跟随，并允许配置
  - 支持显式“回到当前歌词”
- 对外把播放模式收敛成单一枚举，而不是让用户直接理解 `repeat + shuffle`
- 为未来 Web 前端主用的“本地播放模式”预留清晰架构
  - Web 前端多数情况下不依赖 daemon 播放纠偏
  - 数据层与播放层彻底解耦

## 3. 非目标

本次明确不做：

- 不在本次实现完整 Web 前端播放器
- 不把 daemon 播放协议强行改成未来 Web 本地播放器的协议
- 不把用户产品层暴露成“可见队列编辑器”
- 不在本次引入复杂账号同步
- 不把封面做成终端位图缓存系统
- 不在本次实现完整搜索视图、命令面板或日志面板
- 不引入新的播放内核路线
  - 如 GStreamer
- 不在本次把所有旧接口立即删除

## 4. 核心结论

本次采用“共享数据层 + 双播放域 + 轻重状态分流”的方案：

- `共享数据层`
  - 所有客户端共用
  - 提供来源、歌曲、歌词、封面、详情等低频数据
- `远端播放域`
  - 给 daemon/TUI 使用
  - daemon 是播放真相源
  - TUI 是乐观客户端
- `本地播放域`
  - 给未来 Web 前端主用
  - 前端本地持有播放真相
  - daemon 主要提供内容数据，不承担这条播放链路的纠偏

在 daemon/TUI 模式下，再把原来的重聚合模型拆成两层：

- 高频轻状态流
- 低频重内容查询

这样才能同时解决：

- 切歌卡顿
- 底栏更新不顺
- 歌词高亮与自动跟随缺失
- 列表滚动和滚动条缺失
- 当前模型对未来 Web 前端不友好

## 5. 用户模型与播放语义

### 5.1 产品层不暴露“队列”

对用户来说，产品层不存在“队列”概念。

用户真正理解的是：

- 来源
  - 歌单
  - 目录来源
  - 搜索结果来源
- 当前播放
- 播放模式
  - 顺序播放
  - 单曲循环
  - 随机播放
  - 单曲播放

内部仍然允许用稳定播放顺序来实现这些语义，但不在 UI 上把它暴露成“队列编辑器”。

### 5.2 `playlist` 与播放上下文不是一回事

歌单或来源只是内容容器，不等于播放上下文。

当用户在某个来源里选中一首歌并开始播放时，系统应：

- 以该来源的完整内容构造内部播放上下文
- 从该曲开始播放
- 之后的 `next / prev` 作用于这套播放上下文

### 5.3 浏览与播放上下文分离

用户当前浏览的来源，不一定等于当前正在播放的来源。

因此：

- 浏览焦点变化不自动打断播放
- `next / prev` 在 daemon 播放模式下永远作用于当前播放上下文
- 而不是作用于当前浏览焦点

### 5.4 播放模式

对外统一使用单一枚举：

- `ordered`
- `repeat_one`
- `shuffle`
- `single`

内部仍允许暂时保留 `repeat + shuffle` 组合实现，并建立稳定映射：

- `ordered`
  - `repeat = off`
  - `shuffle = false`
- `repeat_one`
  - `repeat = one`
  - `shuffle = false`
- `shuffle`
  - `repeat = off`
  - `shuffle = true`
- `single`
  - 视为只播放当前曲，播完即停

### 5.5 默认模式配置

默认播放模式支持两层来源：

- 全局默认配置
- 客户端记住上次选择

行为规则：

- 若客户端存在上次选择，则优先使用客户端记忆
- 否则退回全局默认值

### 5.6 随机播放语义

随机播放使用“固定随机序列”模型：

- 进入随机模式时，不立刻跳歌
- 当前曲保持不变
- 只对后续顺序进行洗牌
- `prev / next` 在当前随机序列里稳定前进/后退

这样最符合用户预期，也最容易在 daemon 模式与未来本地播放模式之间保持一致。

## 6. 架构总览

### 6.1 共享数据层

共享数据层负责低频内容数据，不带播放真相语义：

- 来源列表
- 来源内歌曲列表
- 歌曲详情
- 歌词原文
- 解析后的歌词时间轴
- 封面引用
- 终端封面摘要
- 默认播放模式配置

这层是未来 TUI 与 Web 前端真正应该长期共享的稳定模型。

### 6.2 远端播放域

daemon 播放模式下：

- daemon 持有播放真相
- TUI 本地做乐观更新
- daemon 用轻量状态流确认或纠偏

这层适用于：

- TUI
- 少量远控场景
- `melo next / prev / toggle`

### 6.3 本地播放域

未来 Web 前端主用：

- 浏览器本地持有播放状态
- 数据仍来自共享数据层
- 不依赖 daemon 的 `play / next / prev` 接口纠偏
- 如需持久化，仅使用 `localStorage` 或后续独立机制

这意味着 Web 前端与 TUI 共用数据模型，但不共用播放真相源。

## 7. daemon/TUI 状态分层

### 7.1 当前问题

当前 `/api/ws/tui` 的聚合快照过重：

- 进度变化是高频事件
- 但每次变化都会带动整页内容重新聚合
- 当前聚合还会牵涉歌词、封面、来源列表等低频数据

这会放大卡顿。

### 7.2 新的双层状态模型

daemon/TUI 模式下，状态拆成两层：

- `PlaybackRuntimeState`
  - 高频
  - 轻量
  - 持续流式推送
- `ContentSnapshot`
  - 低频
  - 按需查询与缓存

### 7.3 高频轻状态流字段

高频流只包含所有客户端都高频关心的字段：

- `generation`
- `playback_state`
- `current_source_ref`
- `current_song_id`
- `current_index`
- `position_seconds`
- `duration_seconds`
- `playback_mode`
- `volume`
- `muted`
- `last_error`

这一层不再承载：

- 歌词正文
- 封面引用
- 来源列表
- 任务栏大块文本

### 7.4 低频内容查询

低频查询单独负责：

- 来源列表
- 来源内歌曲
- 歌词内容
- 封面引用与终端摘要
- 当前歌曲详情
- 手动刷新当前曲内容

## 8. TUI 本地状态模型

### 8.1 两层本地 store

TUI 本地状态拆成：

- `data_cache`
- `ui_state`

### 8.2 `data_cache`

负责缓存低频内容：

- 来源列表缓存
- 当前浏览来源歌曲列表缓存
- `song_id -> track content` 缓存
  - 歌词原文
  - 解析后的歌词时间轴
  - 封面引用
  - 封面摘要

### 8.3 `ui_state`

负责纯客户端交互状态：

- 焦点区域
- 选中索引
- 每个列表的 viewport
- 每个列表的滚动条位置
- 歌词区跟随状态
- 自动恢复倒计时
- hover
- pending optimistic action

### 8.4 手动刷新

歌词/封面缓存采用“强缓存 + 手动刷新”策略：

- 默认不在高频路径里重新取
- 用户可显式刷新当前曲内容
- 刷新时只打掉当前歌曲相关缓存，不清空全局

## 9. TUI 首页信息架构

### 9.1 总体结构

首页采用“双中心”结构：

- 浏览区与当前播放区同等重要
- 布局接近对半

推荐结构：

- 左侧
  - 来源列表
  - 来源内歌曲列表
- 右侧
  - 当前播放摘要
  - 歌词主视区
  - 封面摘要
  - 紧凑底栏

### 9.2 左侧：浏览区

左侧明确只负责浏览与选择来源/歌曲，不直接等价于播放命令。

包含两个垂直区块：

- 来源列表
- 来源内歌曲列表

二者都支持：

- 选中高亮
- 自动跟随选中滚动
- 统一滚动条
- 长标题统一裁剪策略

### 9.3 右侧：当前播放区

右侧负责高频播放体验：

- 当前播放信息
- 当前模式
- 进度
- 歌词
- 封面摘要

其中歌词区是主视区，而不是附属说明区。

### 9.4 底栏

底栏只保留紧凑状态与提示，不承载大段上下文文本。

高频信息应留在上方播放器区，避免底栏频繁抖动和拥挤。

## 10. 列表、长内容与滚动条

### 10.1 统一 viewport 模型

所有可滚动区域统一维护：

- `selected_index`
- `scroll_top`
- `visible_height`
- `follow_selection`

适用区域：

- 来源列表
- 来源内歌曲列表
- 歌词列表

### 10.2 自动跟随选中

当选中项离开当前可视区时：

- 视口自动推进
- 滚动条同步更新

确保不会出现“光标继续往下走，但列表内容不滚”的情况。

### 10.3 长内容策略

不同区域使用不同策略：

- 来源列表 / 歌曲列表
  - 单行显示
  - 默认省略
  - 聚焦行或当前播放行可选横向跑马灯
- 歌词区
  - 按歌词行渲染
  - 超长行软换行
  - 不使用横向跑马灯
- 底栏
  - 紧凑文本
  - 禁止承载超长上下文说明

### 10.4 滚动条

所有长列表统一渲染滚动条：

- 视觉位置一致
- 高度反映当前视口占比
- 滑块位置反映 `scroll_top`

## 11. 歌词区设计

### 11.1 歌词缓存与行模型

歌词以 `song_id + lyrics_source` 为 key 缓存。

解析后形成歌词时间轴行模型：

- `timestamp`
- `text`
- `line_index`
- 可选扩展字段
  - 原始范围
  - 是否为空白间奏行

### 11.2 跟随状态机

歌词区至少有三种状态：

- `FollowCurrent`
  - 自动跟随当前播放行
- `ManualBrowse`
  - 用户手动滚动浏览历史歌词
- `ResumePending`
  - 停止操作后开始倒计时，等待恢复自动跟随

### 11.3 默认恢复规则

当用户手动滚动歌词后：

- 暂停自动跟随
- 默认 `3s` 后恢复自动跟随
- 这个超时时间支持配置

### 11.4 显式动作

提供显式动作：

- `回到当前歌词`

它会：

- 取消手动浏览状态
- 立即恢复自动跟随
- 将当前高亮歌词滚回主视区

### 11.5 高亮逻辑

daemon 只推：

- 当前歌曲 ID
- 当前播放秒数

TUI 根据本地歌词时间轴自己计算：

- 当前高亮行
- 当前主视区定位
- 下一行/上一行可视范围

这样既能提升性能，也与未来 Web 前端的本地播放模型一致。

## 12. 交互模型

### 12.1 乐观交互

在 daemon 播放模式下：

- `next`
- `prev`
- `toggle`
- 切模式
- 在来源内播放当前曲

这些动作都采用乐观交互：

- TUI 先更新本地状态
- 主循环不等待远端请求完成
- daemon 通过轻量状态流确认或纠偏

### 12.2 命令与浏览分离

在 daemon 播放模式下：

- `next / prev` 是播放命令
- 不是浏览动作

浏览列表使用独立焦点和滚动动作。

### 12.3 错误纠偏

daemon 模式下只对真正需要真相源确认的错误做纠偏：

- 切歌失败
- 后端中止
- 模式切换失败
- generation 不一致

不再通过重聚合整页内容来“顺便纠偏”。

## 13. API / WebSocket 设计

### 13.1 Bootstrap 面

保留一个低频启动接口，为 TUI 首屏 hydrate 使用：

- 当前 daemon 播放摘要
- 来源列表
- 当前播放来源
- 默认播放模式配置

它只负责初始化，不承担高频刷新。

### 13.2 共享数据接口

共享数据层提供：

- `list_sources`
- `list_source_tracks`
- `get_track_content`
- `refresh_track_content`
- `get_playback_mode_defaults`

其中 `get_track_content` 返回：

- 歌曲元数据
- 歌词原文
- 解析后的时间轴
- 封面引用
- 终端封面摘要

### 13.3 远端播放轻状态流

新增 daemon 播放模式的轻量运行时 WS 流：

- 高频
- 轻字段
- 可供 TUI 高频消费

### 13.4 命令接口

daemon 模式下保留显式命令接口：

- `play_source_from_track`
- `toggle`
- `next`
- `prev`
- `set_playback_mode`
- `seek`
- `refresh_current_track_content`

### 13.5 兼容策略

本轮不立即删除旧接口：

- `/api/tui/home`
- `/api/ws/tui`

但新 TUI 实现逐步迁到：

- `bootstrap`
- `shared data queries`
- `playback runtime ws`

## 14. daemon 播放链路与 `libmpv`

### 14.1 当前问题

当前 `libmpv` 模式下，切歌仍然偏重：

- 每次 `start_session` 都新建一个 `Mpv` 实例

这不利于：

- `next / prev`
- 高频切歌体验

### 14.2 推荐改动

本轮建议把 `libmpv` 改成“会话复用”模型：

- 后端持有一个可复用的 `Mpv` 实例
- 上下文内切歌时只执行轻操作
  - `loadfile replace`
  - 重置 generation
  - 更新轻状态基线

而不是每次销毁并重建整个 `Mpv` 实例。

### 14.3 区分两类切换成本

daemon 内部明确区分：

- `切播放上下文`
  - 例如从来源 A 切到来源 B
- `上下文内 next / prev`
  - 应尽量轻，只更新索引和 `loadfile`

### 14.4 `PlayerService` 边界

`PlayerService` 继续作为 daemon 播放真相源，但职责收敛为：

- 维护远端播放状态
- 发布轻量运行时状态
- 处理模式切换与错误

它不再在高频路径里负责：

- 查询歌词
- 查询封面
- 重新列来源列表

## 15. 未来 Web 前端模型

未来 Web 前端主打“本地播放模式”：

- 本地播放真相源在浏览器
- 数据来自共享数据层
- 默认不走 daemon 播放命令
- 如需持久化，优先 `localStorage`

这样可以保证：

- TUI 这轮优化不会和未来 Web 架构冲突
- Web 前端不会被当前 daemon 播放协议绑住

## 16. 迁移路径

### Phase 1：客户端状态解耦

- TUI 引入 `data_cache + ui_state`
- 列表 viewport / 滚动条 / 歌词状态机先落地

### Phase 2：协议减重

- 新增 `bootstrap + shared data queries + playback runtime ws`
- TUI 改用新链路

### Phase 3：播放热路径减重

- `libmpv` 会话复用
- 上下文内切歌变轻
- daemon 高频路径不再重查低频内容

### Phase 4：布局与交互替换

- 双中心首页正式替换
- 长标题策略统一
- 手动刷新、回到当前歌词、模式展示等补齐

## 17. 测试策略

### 17.1 协议测试

验证：

- 轻量 WS 只推高频字段
- 重数据查询可独立获取与刷新

### 17.2 本地状态测试

验证：

- 列表自动跟随选中
- 歌词手动滚动后暂停跟随
- 默认 `3s` 恢复自动跟随
- `回到当前歌词`

### 17.3 乐观交互测试

验证：

- `next / prev / toggle` 不阻塞主循环
- 成功时远端确认
- 失败时正确纠偏

### 17.4 `libmpv` 测试

验证：

- 会话复用下 `next / prev` 不重建大对象
- 切来源播放仍能稳定重置上下文

### 17.5 回归体验测试

验证：

- 底栏进度持续更新
- 歌词高亮正确
- 列表选中项不会跑出屏幕
- 所有长列表都有滚动条

## 18. 验收标准

- TUI 在 daemon 播放模式下的 `next / prev / toggle` 不再出现明显阻塞感
- 高频播放状态更新不再依赖整页重聚合快照
- 浏览区与当前播放区构成双中心首页
- 所有长列表都具备统一滚动与滚动条
- 歌词区支持：
  - 自动跟随
  - 当前行高亮
  - 手动回看
  - 默认 `3s` 恢复
  - 手动刷新
- 歌词与封面查一次即可缓存，不在高频路径里重复获取
- `next / prev` 明确只作用于当前播放上下文
- 播放模式对外统一为单一枚举
- 新数据层模型可同时服务未来 Web 前端与当前 TUI

## 19. 结论

本次设计的本质不是“给 TUI 补几个交互细节”，而是把 Melo 正式拆成：

- 稳定共享的数据层
- 少用但可靠的 daemon 远端播放模式
- 未来常用的前端本地播放模式

在这个前提下，TUI 先落地为 daemon 播放模式下的高可用客户端：

- 轻状态流
- 本地缓存
- 乐观交互
- 双中心首页
- 歌词主视区
- 统一滚动模型

这样既能解决当前的卡顿、滚动、歌词体验问题，也不会把未来 Web 前端绑死在当前 daemon 播放协议上。
