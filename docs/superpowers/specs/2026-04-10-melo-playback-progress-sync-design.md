# Melo 下一阶段：播放进度同步设计

日期：2026-04-10

## 1. 目标

在自动下一首能力建立之后，为播放器补齐“时间位置”这条运行时状态。

本阶段的核心目标是：

- 让 `PlayerSnapshot` 能稳定表达当前播放位置
- 让 HTTP / WebSocket / CLI / TUI 读取同一份进度数据
- 控制进度广播频率，避免无意义刷屏
- 为后续会话恢复提供可持久化的位置基线

一句话概括：

把播放器从“知道在播哪首”推进到“知道播到了哪里”。

## 2. 当前现状与问题

当前播放器已经能够表达：

- 播放状态
- 当前歌曲
- 队列位置
- 上一首 / 下一首可用性
- 最近错误

但仍缺少：

- 当前播放位置
- 暂停或停止时的停留位置
- 自然切歌前的最终位置收敛
- 进度更新节流策略

这会导致：

- TUI 无法显示真实进度条
- CLI `status` 只能看到歌曲级状态，不能看到时间位置
- 会话恢复无法知道应该从哪里继续

## 3. 备选方向与推荐

### 3.1 方案 A：由客户端根据“开始播放时间”本地估算

优点：

- service 改动最少
- HTTP 压力最小

缺点：

- CLI、TUI、WebSocket 会各自估算，结果容易漂移
- pause / seek / 自动下一首后会出现不同步
- 无法成为持久化基线

### 3.2 方案 B：由 backend 直接高频推送位置事件

优点：

- 理论上精度高
- TUI 实时感更强

缺点：

- backend 会承担过多广播责任
- 事件频率高，容易让 service 和消费面被动复杂化

### 3.3 方案 C：service 拥有进度时钟，backend 负责提供当前位置

优点：

- 继续保持 `PlayerService` 是唯一状态写入口
- 节流、去抖和广播策略可以统一收口
- 最适合作为恢复与 seek 的前置基础

缺点：

- 需要新增 backend 位置查询能力
- 需要引入 service 内部的定时任务

### 3.4 推荐结论

本阶段选择 **方案 C：service 拥有进度时钟，backend 负责提供当前位置**。

## 4. 本阶段范围

### 4.1 纳入本阶段

- `PlayerSnapshot` 新增进度字段
- backend 提供当前位置查询能力
- service 在播放中定期刷新进度
- pause / stop / track end 时发布最终进度
- WebSocket 广播节流
- CLI / TUI 统一消费进度

### 4.2 不纳入本阶段

- seek
- 歌词时间轴同步
- 波形图或频谱
- 毫秒级精确同步承诺

## 5. 快照契约扩展

建议在 `PlayerSnapshot` 中新增以下字段：

- `position_seconds`
- `position_fraction`

语义建议：

- `position_seconds`：当前播放位置，单位秒；无当前曲目时为 `None`
- `position_fraction`：`0.0 ~ 1.0` 的归一化进度；当总时长未知时为 `None`

本阶段不额外引入独立 `ProgressSnapshot`，避免对外契约拆得过细。

## 6. 模块边界

### 6.1 `src/domain/player/backend.rs`

新增“读取当前位置”的能力，例如：

- `current_position`

这一层只回答“当前 backend 认为播到了哪”，不决定多久广播一次。

### 6.2 `src/domain/player/service.rs`

负责：

- 在 `playing` 状态下启动进度刷新任务
- 在 `paused / stopped / error / idle` 时停止或冻结刷新
- 只在位置变化达到阈值时发布新快照
- 在切歌、暂停、停止时先刷一次最终位置

### 6.3 `src/api/ws.rs`

仍然只广播 `PlayerSnapshot`。

也就是说，WebSocket 不需要新增专门的“progress channel”，保持契约单一。

## 7. 进度更新语义

### 7.1 刷新周期

本阶段建议使用固定节流周期，例如：

- 250ms 到 500ms 之间

推荐基线：

- 默认 500ms

原因：

- 对 TUI 进度条已足够流畅
- 能明显降低高频广播压力
- 仍保留后续调优空间

### 7.2 何时发布新快照

只有在以下情况才发布：

- 位置变化超过阈值
- 播放状态发生变化
- 切换到新歌曲
- 自然结束收敛前的最终位置

不应因为定时器触发但位置未变化，就重复 bump `version`。

### 7.3 暂停与停止

语义如下：

- `pause`：保留当前位置，并停止继续推进
- `stop`：保留当前位置或归零，必须统一规则

推荐规则：

- `pause` 保留当前位置
- `stop` 归零为当前曲目的开头

原因：

- `pause` 是临时中断，用户预期可继续
- `stop` 是显式结束，用户预期下次重新开始

### 7.4 切歌与自然结束

语义如下：

- 切到下一首时，进度立即重置到 `0`
- 队尾自然结束时，最终进度应收敛到接近总时长
- 自动下一首成功后，新歌进度从 `0` 开始

## 8. CLI / TUI / API 影响

### 8.1 CLI

`melo status` 与 `melo queue show` 直接输出新的进度字段。

不要求本阶段为 CLI 新增专门的进度命令。

### 8.2 TUI

TUI 继续只消费快照，但可以开始安全展示：

- 进度条
- 已播时间 / 总时长
- 暂停态下的静止位置

### 8.3 API / WebSocket

- `GET /api/player/status` 返回带进度的快照
- `WS /api/ws/player` 按节流频率广播更新后的快照

## 9. 测试策略

### 9.1 service 单元测试

必须覆盖：

- `playing` 状态下进度递增
- `paused` 后进度不再变化
- `stop` 后进度归零
- 切歌后新曲目进度从 `0` 开始
- 定时刷新但位置未变时不 bump `version`

### 9.2 fake backend 测试

需要能够模拟：

- 连续位置推进
- 固定位置不动
- 查询失败

### 9.3 集成测试

必须覆盖：

- WebSocket 能收到节流后的进度更新
- HTTP `status` 与 WebSocket 看到的当前位置一致

## 10. 验收标准

本阶段完成时，应满足以下条件：

- `PlayerSnapshot` 能稳定表达当前位置
- CLI / HTTP / WebSocket / TUI 共享同一份进度数据
- 暂停、停止、切歌时进度语义清晰一致
- 进度广播有节流，不会无意义刷屏
- 进度更新只在真正变化时递增 `version`

## 11. 与后续阶段的关系

本阶段完成后：

- 会话恢复可以直接持久化当前位置
- 未来 seek 可以在同一套位置契约上扩展
- 歌词同步也可以建立在同一份时间基线上
