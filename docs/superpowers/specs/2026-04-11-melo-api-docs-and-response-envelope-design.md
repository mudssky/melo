# Melo API 文档与统一响应封装设计

日期：2026-04-11

## 1. 目标

本设计聚焦于为 Melo 的 daemon HTTP API 增加正式 API 文档能力，并在接入 React 与其他前端之前，先统一服务端响应契约。

本阶段目标是：

- 为现有 HTTP API 建立正式 OpenAPI 文档
- 提供在线可浏览的 API 文档页面
- 支持导出静态 OpenAPI 文件，便于前端消费与 CI 校验
- 将 HTTP 响应统一为 `code/msg/data` 结构
- 建立稳定的业务错误码体系
- 让 Rust CLI 客户端统一解包响应，不再散落处理协议细节

一句话概括：

把 Melo 当前“可调用但缺少正式契约”的 API，升级成“可文档化、可前端复用、可稳定演进”的接口层。

## 2. 当前现状与问题

当前仓库已经具备以下基础：

- daemon 已暴露多个 HTTP API
- 路由集中定义在 `src/daemon/server.rs`
- 主要 handler 分布在 `src/api/`
- `Cargo.toml` 已引入 `utoipa`
- Rust CLI 已通过 `reqwest` 调用 daemon API

当前主要问题是：

- 没有正式 OpenAPI 文档与可视化入口
- 成功响应格式不统一
- 错误响应没有统一结构
- 多个 handler 直接 `unwrap()`，错误语义无法稳定传递给调用方
- Rust CLI 客户端目前按“裸业务对象”读取响应，不利于后续统一前后端契约
- 后续接入 React 或其他前端时，缺少可依赖的稳定响应协议

## 3. API 风格定位

本项目当前 API 风格整体更接近“命令型 HTTP API”，而不是纯 RESTful 资源接口。

例如：

- `/api/player/play`
- `/api/player/pause`
- `/api/player/next`
- `/api/queue/clear`

这类接口以“触发命令并返回最新状态”为核心，更适合当前 Melo 作为播放器 daemon 控制面的性质。

因此本阶段不把“改造成 RESTful”作为目标，而是明确定位为：

- 命令型 HTTP API
- 局部包含资源查询接口
- 文档上诚实表达其命令语义

这样既保持现有调用方式直观，也避免为了形式统一而做高成本改路由。

## 4. 统一响应契约

### 4.1 HTTP 响应结构

本阶段所有 HTTP 接口统一返回：

```json
{
  "code": 0,
  "msg": "ok",
  "data": {}
}
```

统一响应模型定义为：

- `code: i32`
- `msg: String`
- `data: Option<T>`

### 4.2 成功响应约定

成功时固定约定为：

- `code = 0`
- `msg = "ok"`
- `data` 为实际业务返回值

对于没有业务返回值的接口：

- `data = null`

### 4.3 失败响应约定

失败时同样使用相同外层结构：

```json
{
  "code": 1201,
  "msg": "当前队列为空",
  "data": null
}
```

失败时约定为：

- `code != 0`
- `msg` 为稳定、可面向前端展示或映射的错误说明
- `data = null`

### 4.4 HTTP 状态码与业务码并存

本阶段不会因为引入 `code/msg/data` 就全部返回 `200`。

仍保留 HTTP 状态码表达协议层语义：

- `200`：成功
- `400`：请求格式错误或参数校验失败
- `404`：资源不存在或不可访问
- `409`：当前状态不允许执行操作
- `500`：服务内部错误

业务错误码用于描述更具体的产品语义，HTTP 状态码用于表达大类结果。

### 4.5 WebSocket 处理边界

WebSocket 消息不强制复用 `code/msg/data`。

原因是：

- WebSocket 属于事件流协议
- 当前接口语义是持续推送播放器状态
- 强行套用 HTTP 响应壳会让消息结构变重且不自然

本阶段仅要求为 WebSocket 单独补充文档说明，不纳入统一 HTTP 响应壳。

## 5. 错误码设计

### 5.1 设计原则

错误码体系需要满足：

- 对前端稳定
- 对后端易维护
- 能与 HTTP 状态码形成稳定映射
- 不按模块硬切大片区间，避免 `queue/open` 等领域过于稀疏

本阶段采用“按错误性质分层”的设计。

### 5.2 错误码分组

- `0`：成功
- `10xx`：请求格式错误
- `11xx`：参数校验错误
- `12xx`：状态冲突或当前不可执行
- `13xx`：资源不存在或不可访问
- `14xx`：预留给认证、授权、访问控制
- `15xx`：服务内部错误

### 5.3 HTTP 状态码映射

- `400` 对应 `10xx`、`11xx`
- `404` 对应 `13xx`
- `409` 对应 `12xx`
- `500` 对应 `15xx`

`14xx` 本阶段不启用，但预留位置固定，不与其他区间混用。

### 5.4 首批错误码草案

- `1001`：请求体无效
- `1002`：JSON 解析失败
- `1101`：音量超出范围
- `1102`：队列索引无效
- `1103`：循环模式非法
- `1201`：当前队列为空
- `1202`：当前状态下不能执行该操作
- `1301`：打开目标不存在
- `1302`：不支持的目标类型
- `1303`：daemon 不可用或未发现
- `1501`：播放后端异常
- `1502`：数据库异常
- `1503`：文件系统异常
- `1599`：未分类内部错误

本阶段只要求建立“首批最小可用错误码”，不追求一次性穷举所有错误。

## 6. 服务端实现设计

### 6.1 统一响应模型

在 HTTP 边界引入统一响应模型与错误响应模型，供所有 handler 复用。

统一响应壳只存在于 API 边界，不向 domain/service 层渗透。

这样可以保持内部业务逻辑仍然使用原有 `MeloResult<T>` 或等价错误流，而 HTTP 适配层负责：

- 成功时包装 `ApiResponse<T>`
- 失败时映射到统一错误响应

### 6.2 handler 错误处理

当前多个 handler 直接 `unwrap()`，本阶段应改为统一错误返回。

目标是：

- handler 返回可传播错误的结果类型
- 由一层公共错误转换逻辑生成 `HTTP status + ApiResponse<()>`
- 避免每个接口重复写状态码与错误响应构造逻辑

### 6.3 错误映射来源

错误映射优先基于项目内部统一错误类型进行转换。

如果现有 `MeloError` 不能表达足够的 API 语义，本阶段允许增加面向 API 的错误分类或转换层，但不要求大规模重构全部内部错误模型。

### 6.4 保持业务对象不变

本阶段不改变 `PlayerSnapshot`、`HealthResponse`、`DaemonStatusResponse` 等业务对象本身的字段结构。

变化发生在最外层 HTTP 响应壳，而不是侵入业务数据结构。

## 7. Rust CLI 客户端处理设计

### 7.1 不依赖类似 Axios 的内建拦截器

`reqwest` 自身没有像 Axios 那样的内建全局响应拦截器。

因此本阶段采用更 Rust 风格的方式：

- 在 `src/cli/client.rs` 中封装统一请求方法
- 统一发送请求
- 统一解析 `ApiResponse<T>`
- 统一将 `code != 0` 转换为项目错误类型

### 7.2 客户端通用处理路径

客户端通用方法负责：

- 发起请求
- 处理网络错误
- 处理 HTTP 状态码错误
- 反序列化统一响应壳
- 当 `code != 0` 时转换成稳定的 CLI 错误
- 返回解包后的 `T`

上层调用方继续面向“业务对象”编程，而不是在每个 API 方法中重复解析 `code/msg/data`。

### 7.3 中间件边界

本阶段不引入 `reqwest-middleware` 一类额外依赖来处理业务响应壳。

原因是：

- 当前需求核心是业务层统一解包，而不是复杂请求中间件链
- 引入额外依赖会增加实现和维护成本
- 现阶段在 `ApiClient` 中封装已足够满足统一处理目标

未来若需要：

- tracing
- retry
- 全局认证 header
- 更复杂的请求埋点

再评估是否引入 middleware 体系。

## 8. OpenAPI 与在线文档设计

### 8.1 OpenAPI 生成方式

本阶段以 `utoipa` 为核心，为现有 HTTP API 补充：

- 路由说明
- 请求体 schema
- 成功响应 schema
- 错误响应 schema
- 状态码说明

OpenAPI 文档中的响应结构应真实反映统一响应壳，而不是继续描述裸业务对象。

### 8.2 在线文档暴露方式

daemon 启动后应提供：

- OpenAPI JSON 端点，例如 `/api/openapi.json`
- 文档浏览页面，例如 `/api/docs`

具体 UI 方案可以在实现时根据 `utoipa` 生态选择，但本设计要求：

- 本地启动 daemon 后即可直接访问
- 页面内容与 OpenAPI JSON 共用同一份规范来源

### 8.3 文档范围

本阶段文档覆盖：

- `system` HTTP API
- `player` HTTP API
- `queue` HTTP API
- `open` HTTP API

WebSocket 至少补充接口入口和消息说明，但不要求做成与 HTTP 完全同级的完整 schema 体系。

## 9. `package.json` 脚本设计

### 9.1 目标

需要在 `package.json` 中增加与 API 文档相关的统一入口，便于开发者、前端和 CI 使用。

### 9.2 建议脚本

建议新增：

- `docs:api`
  - 生成或刷新 OpenAPI 导出文件
- `docs:api:check`
  - 校验仓库中的 OpenAPI 导出是否为最新
- `docs:api:serve`
  - 提示或辅助开发者访问本地在线文档入口

脚本命名保持与现有项目风格一致，实际底层可以调用 Rust 侧导出命令、Node 辅助脚本或其他仓库内既有工具。

### 9.3 静态导出用途

静态导出的 OpenAPI 文件主要服务于：

- 前端类型生成
- 接口变更审查
- CI 校验
- 离线查看

本阶段只要求导出规范文件，不要求额外生成前端 SDK。

## 10. 本阶段实现边界

### 10.1 本阶段必须完成

- 建立统一 HTTP 响应壳 `code/msg/data`
- 建立首批错误码与 HTTP 状态映射
- 去掉 API handler 中关键路径的直接 `unwrap()`
- 为现有 HTTP API 补 OpenAPI 描述
- 暴露在线文档页面与 OpenAPI JSON
- 在 `package.json` 增加 API 文档导出与校验脚本
- 更新 Rust CLI 客户端以统一解包响应

### 10.2 本阶段不做

- 不重构现有 API 为纯 RESTful 风格
- 不让 WebSocket 强制使用 `code/msg/data`
- 不引入鉴权能力，但保留 `14xx`
- 不生成前端 SDK
- 不把 API 响应壳侵入 domain/service 内部

## 11. 测试策略

本阶段应至少覆盖以下验证：

- HTTP 成功响应统一带有 `code/msg/data`
- 典型失败路径返回正确 HTTP 状态码与业务错误码
- Rust CLI 客户端能正确解包成功响应
- Rust CLI 客户端能正确识别非零 `code`
- OpenAPI 导出结果可生成并稳定校验
- 在线文档入口可访问

若已有 API 测试存在，应优先更新现有测试而不是新增重复覆盖。

## 12. 验收标准

本阶段完成时，应满足以下条件：

- 所有 HTTP API 对外响应结构统一
- `code = 0` 固定表示成功
- 前端可基于统一外层结构接入 Melo API
- 关键错误路径具备稳定错误码
- 仓库可导出并校验 OpenAPI 文档
- 启动 daemon 后可访问在线 API 文档页面
- Rust CLI 客户端无需在各业务方法中重复解析协议细节

## 13. 后续扩展点

本设计完成后，可以自然继续推进：

- 为 WebSocket 定义更正式的事件 schema
- 基于 OpenAPI 生成前端类型
- 增加认证与授权体系并启用 `14xx`
- 为错误码补充更完整的前端展示映射
- 在客户端层引入更完整的 tracing、retry 与认证中间件
